树：倒着长的家族谱

上一节课我们学了链表——小朋友们手拉手排成一队。今天我们要学的数据结构可就厉害了，它让小朋友不再排成一条线，而是分叉、分叉、再分叉，像一棵树一样伸展开来。

开场白

想象一下你打开家谱。你的上面是父母，父母的上面是爷爷奶奶、外公外婆……再往上是祖宗十八代。你会发现：一个人可能有多个后代，但只有一个父母（生物学意义上的，别跟我杠）。

再想想你们公司的组织架构图：CEO在最上面，下面有几个VP，每个VP下面有总监，总监下面有经理，经理下面有你。

这两种结构有个共同点：从上到下，不断分叉，而且没有环。在CS的世界里，我们把这种结构叫做——树（Tree）。

💡 思考问题1：链表是"一条线"，树是"分叉线"。如果让你设计一个能存家谱的数据结构，你会怎么做？

💡 思考问题2：为什么公司的组织架构图不能出现环？如果A管B，B管C，C管A，会发生什么？

核心概念

什么是树？

树是一种非线性的数据结构，它由节点（node）和边（edge）组成。

简单说：树就是一堆点，点之间用线连起来，而且没有环。

更正式一点：

  • 树要么为空（没有节点）
  • 要么有一个根节点（root），以及若干互不相交的子树（subtree）

这个定义本身就很"递归"——"子树本身也是一棵树"。套娃文学在CS中屡见不鲜。

树的基本术语

术语 含义 生活类比
根（Root） 树的最顶层节点 家谱里的老祖宗、公司CEO
叶子（Leaf） 没有子节点的节点 家谱最底层的你、公司基层员工
节点（Node） 树中的每个元素 家谱中的每个人
边（Edge） 两个节点之间的连线 家谱中的父子关系线
父节点（Parent） 当前节点的直接上级 你爸
子节点（Child） 当前节点的直接下级 你儿子
兄弟（Sibling） 同一个父节点的子节点 你亲兄弟
深度（Depth） 从根到该节点的边数 你是第几代子孙
高度（Height） 从该节点到最远叶子的边数 你的后代最多还能传几代
度（Degree） 一个节点的子节点个数 你有几个孩子

一个容易混淆的点：树的"根"在上面，"叶子"在下面——这跟现实中的树是倒过来的！ 所以CS的树是"倒着长的"。这是历史原因，早期数学家画树就是这么画的，大家就跟着这么叫了。

二叉树

二叉树是每个节点最多有两个子节点的树——一个左孩子，一个右孩子。

       A
     /   \
    B     C
   / \   / \
  D   E F   G

这就是二叉树的经典画。每个节点最多牵着两根线。

二叉树有几个特殊类型：

类型 定义 特点
满二叉树 所有层的节点都满了 第k层有2^k个节点
完全二叉树 除了最后一层，其他层都满；最后一层从左到右排列 堆就是完全二叉树
完美二叉树 所有叶子在同一层，所有非叶子都有两个子节点 满二叉树+叶子都在最底层

二叉树的几个重要性质：

  • 第i层最多有 2^(i-1) 个节点（根在第1层）
  • 高度为h的二叉树最多有 2^h - 1 个节点
  • 叶子节点数 = 度为2的节点数 + 1

# code/examples/example-01-binary-tree.py

运行结果：

=== 二叉树基本属性 ===

满二叉树高度: 3

满二叉树性质验证: 2^3 - 1 = 7 (实际节点数: 7)

你看，二叉树就是TreeNode对象互相指向对方。每个节点最多有两个"小弟"——左小弟和右小弟。高度和节点数的计算都用了递归——大树的问题，可以拆成小树的问题。

示例2：四种遍历方式

树怎么"逛"一遍？这可比链表复杂多了——链表只有一个方向"往前走"，但树有分叉！

四种遍历方式，口诀如下：

# code/examples/example-02-traversals.py

运行结果：

=== 测试树 ===
         1
       /   \
      2     3
     / \   /
    4   5 6

前序遍历（根-左-右）: [1, 2, 4, 5, 3, 6]
中序遍历（左-根-右）: [4, 2, 5, 1, 6, 3]
后序遍历（左-右-根）: [4, 5, 2, 6, 3, 1]
层序遍历（逐层）:     [[1], [2, 3], [4, 5, 6]]

=== 遍历速记口诀 ===
前序：根左右 —— 先打卡，再逛两边
中序：左根右 —— 从左开始，依次路过
后序：左右根 —— 先逛完，最后回来
层序：一层层 —— 从上到下，从左到右

为什么要有四种遍历？ 因为不同的场景需要不同的访问顺序：

  • 前序遍历：复制一棵树。先造根，再造左右。就像盖房子先打地基再盖上面。
  • 中序遍历：BST中序遍历会得到有序序列。这是BST的核心性质。
  • 后序遍历：删除一棵树。先删子树，再删根。就像拆房子从顶往下拆。
  • 层序遍历：BFS搜索、求树的宽度。一层一层扫过去。

BST是树家族里最有"实用性"的成员——它自带排序属性。

# code/examples/example-03-binary-search-tree.py

运行结果：

=== BST 构建完成 ===
插入的值: [8, 3, 10, 1, 6, 14, 4, 7, 13]
中序遍历（有序输出）: [1, 3, 4, 6, 7, 8, 10, 13, 14]

=== 查找 ===
查找 6: 找到
查找 99: 未找到

=== 删除 ===
删除 4 后: [1, 3, 6, 7, 8, 10, 13, 14]
删除 3 后: [1, 6, 7, 8, 10, 13, 14]
删除 10 后: [1, 6, 7, 8, 13, 14]

BST的精髓就是：左 < 根 < 右。这使得中序遍历天然有序。查找、插入、删除都是O(h)——h为树的高度。

深入理解

BST删除的"魔术操作"

删除BST中一个有两个孩子的节点时，为什么用右子树的最小值（中序后继）来替换？

想一下：我们要找一个值来替换被删除的节点，这个值必须满足：

  1. 比左子树所有值都大
  2. 比右子树所有值都小

右子树的最小值正好满足这两个条件！它就在右子树最左边的那个节点。

同理，用左子树的最大值（中序前驱）也可以，效果等价。选哪个都行，一般习惯用后继。

树的退化问题

BST有个致命缺陷：如果插入的数据是有序的，BST会退化成一个链表！

正常BST：          退化BST（有序插入）：
      5                  1
    /   \                  \
   3     7                  2
  / \   / \                  \
 1   4 6   8                  3
                                 \
                                  4
                                   \
                                    5

左边高度为2，查找最多比较2次。右边高度为5，查找要比较5次。本来O(log n)的操作变成了O(n)。

这就是为什么工业界使用平衡二叉搜索树（AVL树、红黑树、B树等）来保证树的高度是O(log n)。下节课我们可能会聊这个话题。

时间/空间复杂度对比

操作 数组 链表 BST（平衡） BST（退化）
查找 O(1) 按索引 / O(n) 按值 O(n) O(log n) O(n)
插入 O(n) O(1) O(log n) O(n)
删除 O(n) O(1) O(log n) O(n)
有序遍历 O(n log n) 排序 O(n) O(n) 中序 O(n)
空间 O(n) 连续 O(n) 非连续 O(n) O(n)

BST的甜区：你需要有序数据 + 高效查找/插入/删除。数组快在按索引访问，但按值查找是O(n)。链表插入快，但查找也是O(n)。BST在两者之间找到平衡。

实际应用

1. 文件系统

你的电脑文件系统就是一棵树：

/
├── Users/
│   └── slouch/
│       ├── Documents/
│       │   └── learn/
│       └── Downloads/
├── Applications/
├── System/
└── tmp/
  • 根目录 / 是树的根
  • 每个文件夹是一个节点
  • 文件是叶子节点
  • 路径 Users/slouch/Documents 就是从根到一个节点的路径

cd 就是往下走一层，ls 就是看当前节点的所有子节点。

2. HTML DOM树

每个网页都是一个DOM树：

<html>
├── <head>
│   ├── <title>
│   └── <meta>
└── <body>
    ├── <h1>
    ├── <p>
    └── <div>
        ├── <span>
        └── <a>

浏览器的渲染引擎从DOM树的根开始遍历，逐层渲染页面。CSS选择器本质上就是在树中搜索节点。

3. 数据库索引（B树/B+树）

数据库索引的核心是B+树——BST的进化版。B+树是一种多路搜索树，每个节点可以有多个孩子，而且所有叶子在同一层。

为什么不用BST？ 因为磁盘IO是按"块"读取的。B+树的一个节点大小和磁盘块对齐，一次磁盘IO可以读取整个节点，效率更高。

MySQL、PostgreSQL等关系型数据库的索引底层都是B+树。这也是为什么你的SQL查询加了索引后，从O(n)变成了O(log n)。

4. 决策树

机器学习中的决策树，每个内部节点是一个判断条件，每个叶子节点是一个结论：

          今天下雨？
         /         \
      是            否
     /               \
  带伞             有太阳？
                  /       \
                是         否
               /           \
           涂防晒霜      正常出门

常见误区

⚠️ 注意这些坑！

误区1：以为BST总是O(log n)

错！ BST只有在平衡的情况下才是O(log n)。有序插入会让BST退化成链表，所有操作变成O(n)。

✅ 正确做法：在需要保证性能的场景，使用平衡BST（AVL树、红黑树、Treap等）。Python的dict底层是哈希表而非BST，但Java的TreeMap底层是红黑树。

误区2：搞混深度和高度

这是新手最容易晕的地方：

  • 深度：从根往下数（根的深度=0）
  • 高度：从叶子往上数（叶子的高度=0）

口诀：深度从上往下跳，高度从下往上爬。一个节点的深度 = 它到根的边数，高度 = 它到最远叶子的边数。

误区3：前序/中序/后序记混

别死记硬背！记住一个核心规则：前中后指的是"根"什么时候被访问。

  • 前序：根前面访问（根-左-右）
  • 中序：根中间访问（左-根-右）
  • 后序：根后面访问（左-右-根）

左右永远是左在前、右在后。变化的只是根的位置。

动手实践

练习1：求二叉树的最大深度 ⭐

描述：给定一棵二叉树，求它的最大深度（根到最远叶子节点的最长路径上的节点数）。

要求：

  • 实现 max_depth(root) 函数
  • 使用递归方法

提示：一棵树的最大深度 = max(左子树最大深度, 右子树最大深度) + 1

💡 提示（点击展开）

递归终止条件：如果节点为空，深度为0。
否则返回 1 + max(max_depth(root.left), max_depth(root.right))

练习2：验证二叉搜索树 ⭐⭐

描述：给定一棵二叉树，判断它是否是合法的BST。

要求：

  • 实现 is_valid_bst(root) 函数
  • 合法BST：左子树所有节点值 < 根节点值 < 右子树所有节点值

提示：中序遍历BST会得到有序序列。如果中序遍历结果不是递增的，就不是合法BST。

💡 提示（点击展开）

方法1：中序遍历后检查结果是否有序。
方法2：递归时维护上下界，每个节点的值必须在(min, max)范围内。

练习3：二叉树的最近公共祖先 ⭐⭐⭐

描述：给定一棵二叉树和两个节点p、q，找到它们的最近公共祖先（LCA）。

要求：

  • 实现 lowest_common_ancestor(root, p, q) 函数
  • 最近公共祖先：是p和q的祖先，且在所有公共祖先中深度最深

提示：从根开始递归。如果p和q分别在当前节点的左右两侧，当前节点就是LCA。

💡 提示（点击展开）

递归思路：

  1. 如果当前节点是p或q，返回当前节点
  2. 递归查找左子树和右子树
  3. 如果左右子树都找到了节点 → 当前节点是LCA
  4. 如果只有左子树找到 → LCA在左子树
  5. 如果只有右子树找到 → LCA在右子树

小结与思考

本节要点

  1. 树是分层结构：根在最上面，叶子在最下面，每个节点可以有多个子节点
  2. 二叉树每个节点最多两个孩子：左孩子和右孩子，有序号之分
  3. 四种遍历方式：前序（根左右）、中序（左根右）、后序（左右根）、层序（逐层）
  4. BST的核心性质：左 < 根 < 右，中序遍历天然有序
  5. BST的查找/插入/删除：平均O(log n)，最坏O(n)（退化时）
  6. 树的应用无处不在：文件系统、DOM树、数据库索引、决策树

思考题

  1. 如果让你用数组来实现一棵完全二叉树，怎么设计？ 提示：根在索引0，节点i的左孩子在2i+1，右孩子在2i+2。这是堆的实现方式，想想为什么可以这样做？

  2. BST的删除操作中，为什么可以用左子树的最大值代替右子树的最小值？ 这两种方式有什么优劣？在什么场景下应该选哪种？

  3. 如果我们要频繁地查找"第k大的元素"，BST够好吗？ 可能需要对BST做怎样的改造？
