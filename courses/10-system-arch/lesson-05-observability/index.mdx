## 开场白

在现代分布式系统中，应用通常由数十甚至数百个微服务组成，跨越多个数据中心和云平台。当用户遇到问题时，传统的调试方法往往力不从心——你无法简单地"登录服务器查看日志"，因为请求可能经过了多个服务，每个服务都有自己的日志和指标。

可观测性（Observability）正是解决这一挑战的关键能力。它不仅仅是监控的升级版，而是一种系统性的方法论，让我们能够理解复杂系统的内部状态，即使面对从未见过的问题。本课程将深入探讨可观测性的三大支柱：**日志（Logs）**、**指标（Metrics）** 和 **链路追踪（Traces）**，并展示如何将它们有机结合，构建完整的可观测性体系。

通过本课程的学习，你将掌握：
- 如何设计结构化的日志记录策略
- 如何收集和分析关键业务指标  
- 如何实现跨服务的分布式追踪
- 如何将三者关联起来进行高效的问题排查

## 核心概念

### 什么是可观测性？

可观测性源于控制理论，指的是通过系统的外部输出来推断其内部状态的能力。在软件工程中，可观测性意味着我们能够回答这样的问题："为什么系统表现出这种行为？"

与传统的监控不同，可观测性强调**探索性分析**而非预定义的告警。监控告诉你"什么坏了"，而可观测性帮助你理解"为什么坏了"。

### 三大支柱详解

#### 1. 日志（Logs）
日志是离散的事件记录，通常是文本格式。它们提供了系统运行的详细叙事，包括：
- **结构化日志**：使用 JSON 等格式，便于机器解析
- **上下文信息**：包含 Trace ID、用户 ID、请求 ID 等关联标识
- **日志级别**：DEBUG、INFO、WARNING、ERROR、CRITICAL

#### 2. 指标（Metrics）
指标是数值型的时间序列数据，用于量化系统状态：
- **Counter**：单调递增的计数器（如请求总数）
- **Gauge**：可增可减的瞬时值（如内存使用量）
- **Histogram**：分布统计（如请求延迟分布）
- **Summary**：聚合统计（如平均值、分位数）

#### 3. 链路追踪（Traces）
链路追踪记录单个请求在分布式系统中的完整路径：
- **Trace**：完整的请求链路，由多个 Span 组成
- **Span**：单个操作的执行记录，包含开始时间、持续时间、标签等
- **上下文传播**：通过 HTTP 头部等方式在服务间传递 Trace ID

### 相关标准和工具

- **OpenTelemetry**：CNCF 主导的可观测性标准，统一了 API 和 SDK
- **Prometheus**：开源的监控和告警系统，指标收集的事实标准
- **Jaeger/Zipkin**：分布式追踪系统
- **W3C Trace Context**：标准化的上下文传播格式

### 示例 1：结构化日志

```python
# example-01-structured-logging.py
import json
from datetime import datetime

class StructuredLogger:
    def _format_log_entry(self, level: str, message: str, **kwargs) -> str:
        log_entry = {
            "timestamp": datetime.utcnow().isoformat() + "Z",
            "level": level,
            "message": message,
            "logger": self.logger.name,
            **kwargs
        }
        return json.dumps(log_entry, ensure_ascii=False)
```

这个示例展示了如何创建结构化的 JSON 日志，包含时间戳、级别、消息和任意上下文信息。

## 深入理解

### 为什么需要三者结合？

单独使用任何一种数据源都有局限性：

- **仅用日志**：难以发现模式和趋势，查询效率低
- **仅用指标**：缺乏详细的上下文信息，无法进行根本原因分析  
- **仅用链路追踪**：采样率限制导致可能错过重要问题

三者结合可以相互补充：
- 通过 **Trace ID** 关联日志和链路
- 通过 **指标告警** 触发详细的日志和链路分析
- 通过 **日志上下文** 丰富指标和链路的信息

### 数据模型对比

| 特性 | 日志 | 指标 | 链路追踪 |
|------|------|------|----------|
| 数据量 | 高 | 低 | 中 |
| 查询性能 | 慢 | 快 | 中 |
| 上下文丰富度 | 高 | 低 | 高 |
| 存储成本 | 高 | 低 | 中 |
| 实时性 | 高 | 高 | 高 |

### 采样策略

由于可观测性数据量巨大，合理的采样策略至关重要：

- **日志采样**：基于级别或关键字过滤
- **指标采样**：通常不需要采样，本身就是聚合数据
- **链路追踪采样**：基于 Trace ID 的概率采样或智能采样

## 实际应用

### SRE 实践：SLIs、SLOs、SLAs

可观测性是 Site Reliability Engineering (SRE) 的基础：

- **SLI (Service Level Indicator)**：服务质量指标，如错误率、延迟
- **SLO (Service Level Objective)**：目标值，如"99.9%的请求延迟<100ms"
- **SLA (Service Level Agreement)**：与用户的协议，违反时可能有赔偿

通过可观测性数据，我们可以：
1. 定义准确的 SLI
2. 监控 SLO 合规性
3. 在违反 SLA 前主动干预

### 告警最佳实践

有效的告警应该遵循 **Google SRE 告警原则**：

- **症状告警**：基于用户体验的问题，而非原因
- **可操作性**：每个告警都应该有明确的响应流程
- **避免噪音**：合理设置阈值，防止告警疲劳

例如，不要告警"CPU 使用率 > 80%"，而应该告警"API 错误率 > 1%"。

### 成本优化

可观测性数据的存储和处理成本可能很高，需要平衡：

- **日志保留策略**：热数据（7天）、温数据（30天）、冷数据（1年）
- **指标降采样**：高精度数据短期保留，长期使用聚合数据
- **智能采样**：对错误请求 100% 采样，正常请求概率采样

## 常见误区

### 误区 1：可观测性就是监控

虽然相关，但可观测性更强调**探索性分析**能力。监控是预先定义的规则，可观测性支持即席查询和未知问题的排查。

### 误区 2：越多的日志越好

过度的日志记录会导致：
- 存储成本激增
- 查询性能下降  
- 重要信息被噪音淹没

应该记录**有意义的结构化日志**，而不是所有可能的信息。

### 误区 3：链路追踪总是 100% 采样

全量采样在高流量系统中成本过高。应该采用**智能采样策略**：
- 错误请求 100% 采样
- 正常请求按比例采样
- 关键业务路径提高采样率

### 误区 4：指标越多越好

过多的指标会导致：
- 存储和计算成本增加
- 告警配置复杂化
- 团队注意力分散

应该聚焦于**关键业务指标**和**用户体验指标**。

### 误区 5：忽视上下文关联

如果日志、指标、链路追踪之间没有关联标识符（如 Trace ID），就无法进行跨维度分析，大大降低可观测性的价值。
