# 挑战1：设计分布式任务调度系统

## 难度：⭐⭐⭐⭐

## 描述
在微服务架构中，经常需要执行定时任务或延迟任务，比如：
- 每天凌晨2点生成报表
- 订单创建30分钟后自动取消（如果未支付）
- 定期清理过期数据

传统的单机定时任务无法满足高可用和扩展性要求。你需要设计一个分布式的任务调度系统，能够在多个节点间协调任务执行，确保任务不会重复执行，并且在节点故障时能够自动恢复。

## 要求
你的分布式任务调度系统必须实现以下功能：

### 核心功能
1. **任务注册**：支持注册一次性任务和周期性任务
2. **分布式协调**：多个调度节点协同工作，避免任务重复执行
3. **故障恢复**：当某个节点宕机时，其他节点能接管其任务
4. **任务持久化**：任务信息需要持久化存储，防止数据丢失

### 技术约束
- 使用Python标准库实现核心逻辑
- 可以使用文件系统或内存数据结构模拟持久化存储
- 实现简单的领导者选举机制（不需要复杂的共识算法）
- 支持基本的任务状态管理（待执行、执行中、已完成、失败）

### 扩展功能（可选）
- 任务优先级支持
- 任务依赖关系（任务B在任务A完成后执行）
- 任务执行结果回调
- Web界面展示任务状态

## 提示
1. **领导者选举**：可以使用基于心跳的简单选举机制，或者利用文件锁/数据库锁
2. **任务分配**：考虑使用一致性哈希或简单的轮询分配策略
3. **故障检测**：通过定期心跳来检测节点健康状态
4. **数据结构**：设计合理的任务数据结构，包含ID、类型、执行时间、状态等字段
5. **并发控制**：使用线程安全的数据结构和锁机制

## 评估标准
- 系统的正确性（任务不重复执行、故障恢复正常）
- 代码的清晰度和可维护性
- 对分布式系统挑战的理解和处理
- 创新性和扩展性考虑

## 示例场景
```python
# 创建调度器实例
scheduler = DistributedScheduler(node_id="node-1", storage_path="./tasks")

# 注册周期性任务
scheduler.register_task(
    task_id="daily-report",
    task_func=generate_daily_report,
    schedule="0 2 * * *",  # 每天凌晨2点
    is_recurring=True
)

# 注册延迟任务
scheduler.register_task(
    task_id="cancel-order-123",
    task_func=cancel_order,
    args=("order-123",),
    delay=1800  # 30分钟后
)
```

记住，这是一个分布式系统设计挑战，重点在于处理网络分区、节点故障和数据一致性等实际问题。
