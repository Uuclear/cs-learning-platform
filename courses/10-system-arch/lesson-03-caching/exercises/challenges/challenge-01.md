# 挑战 1：实现分布式缓存一致性

## 难度
⭐⭐⭐

## 背景
在分布式系统中，多个服务实例可能同时访问和修改缓存数据，这会导致缓存一致性问题。当一个服务更新了数据库但其他服务的缓存仍然是旧数据时，就会出现不一致。

## 任务要求
实现一个简单的分布式缓存一致性解决方案，使用以下策略之一：

1. **Cache-Aside + 延迟双删**：在更新数据库后，先删除缓存，延迟一段时间后再删除一次
2. **消息队列通知**：使用简单的消息队列模拟，当数据更新时通知所有缓存节点失效
3. **版本号控制**：为每个数据项添加版本号，缓存存储版本号，读取时验证版本

## 具体实现
创建一个 `DistributedCacheConsistency` 类，包含以下方法：

- `get(key: str) -> Optional[Any]`：获取数据
- `update(key: str, value: Any) -> None`：更新数据并处理一致性
- `invalidate_cache(key: str) -> None`：使缓存失效（用于消息通知场景）

## 测试场景
模拟两个服务实例同时操作同一数据：
1. 实例A读取数据（缓存未命中，从数据库加载）
2. 实例B更新数据
3. 实例A再次读取数据（应该得到最新数据）

## 提示
- 可以使用字典模拟数据库和缓存
- 使用 `time.sleep()` 模拟网络延迟
- 考虑并发安全问题（虽然Python GIL简化了问题，但仍需考虑逻辑正确性）

## 评估标准
- 正确处理缓存一致性
- 代码结构清晰，有适当的注释
- 处理边界情况（如缓存未命中、数据库异常等）
