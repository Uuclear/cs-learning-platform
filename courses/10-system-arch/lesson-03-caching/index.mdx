## 开场白

想象一下，你正在运营一个热门的电商平台。在双十一大促期间，你的网站每秒要处理成千上万的用户请求。每个用户都要查看商品详情、价格、库存等信息。如果没有缓存，每次请求都要查询数据库，数据库很快就会不堪重负，响应时间从毫秒级变成秒级，甚至直接宕机。

这就是我们今天要解决的问题——缓存。通过在内存中存储热点数据，我们可以将数据库的负载降低90%以上，同时将响应时间从几十毫秒降低到几毫秒。缓存不仅提升了用户体验，还大幅降低了服务器成本。

在这个课程中，我们将深入探讨缓存的核心概念、常用模式、以及如何使用Redis和Memcached构建高效的缓存系统。

## 核心概念

### 缓存模式

**Cache-Aside（旁路缓存）**：应用直接管理缓存和数据库。读取时先查缓存，未命中再查数据库；写入时先更新数据库，再删除缓存。这是最常用的模式，简单可靠。

**Read-Through（读穿透）**：对应用透明，缓存层自动处理缓存未命中的情况，从数据库加载数据并缓存。

**Write-Through（写穿透）**：写入时同时更新缓存和数据库，保证数据一致性，但写入延迟较高。

### 缓存失效策略

**TTL（Time To Live）**：为缓存设置过期时间，自动清理过期数据。

**LRU（Least Recently Used）**：当缓存满时，淘汰最久未使用的数据。

**LFU（Least Frequently Used）**：淘汰使用频率最低的数据。

**主动失效**：当数据更新时，主动删除或更新缓存。

### 缓存问题及解决方案

**缓存穿透**：查询不存在的数据，导致每次都要查询数据库。解决方案：缓存空值或使用布隆过滤器。

**缓存雪崩**：大量缓存同时过期，导致数据库瞬间压力剧增。解决方案：设置随机过期时间。

**缓存击穿**：热点数据过期时，大量请求同时打到数据库。解决方案：互斥锁或永不过期+后台更新。

**热点Key问题**：某个Key被频繁访问，单个缓存节点压力过大。解决方案：本地缓存+多级缓存。

## 深入理解

### 分布式缓存一致性

在分布式环境中，多个服务实例可能同时修改同一份数据，这会导致缓存不一致。常见的解决方案包括：

1. **最终一致性**：允许短暂的不一致，通过定时同步或消息通知来修复
2. **强一致性**：使用分布式锁或事务来保证一致性，但会牺牲性能
3. **版本控制**：为数据添加版本号，读取时验证版本

### Redis 内部实现

Redis 的高性能来源于以下几个关键设计：

- **单线程事件循环**：避免了多线程的上下文切换开销
- **内存存储**：所有数据都在内存中，访问速度极快
- **多种数据结构**：针对不同场景优化的数据结构（字符串、哈希、列表、集合、有序集合）
- **IO多路复用**：使用 epoll/kqueue 处理大量并发连接
- **持久化机制**：RDB 快照和 AOF 日志保证数据不丢失

Redis 的有序集合（Sorted Set）使用跳跃表（Skip List）实现，既保证了 O(log N) 的查询性能，又支持范围查询。

### Memcached vs Redis

| 特性 | Memcached | Redis |
|------|-----------|-------|
| 数据结构 | 仅字符串 | 字符串、哈希、列表、集合、有序集合 |
| 持久化 | 不支持 | 支持 RDB 和 AOF |
| 集群 | 客户端分片 | 原生集群支持 |
| 内存管理 | Slab分配器 | 自定义内存分配器 |
| 多线程 | 多线程 | 单线程（6.0+支持多线程IO） |

选择建议：
- 简单的键值缓存 → Memcached
- 复杂数据结构、持久化需求 → Redis
- 高并发简单操作 → Memcached
- 功能丰富、灵活使用 → Redis

## 实际应用

### Session 存储
使用 Redis 存储用户会话信息，支持分布式部署和会话共享：
```python
# 设置会话
redis.setex(f"session:{session_id}", 3600, user_data)

# 获取会话  
session_data = redis.get(f"session:{session_id}")
```

### 限流控制
使用 Redis 的 INCR 和 EXPIRE 命令实现简单的计数器限流。通过维护一个按时间排序的请求记录列表，可以实现滑动窗口算法。

### 排行榜
使用 Redis 的 Sorted Set 实现实时排行榜：
```python
# 更新分数
redis.zadd("leaderboard", {user_id: score})

# 获取排名前10
top_10 = redis.zrevrange("leaderboard", 0, 9, withscores=True)
```

### 缓存预热
在系统启动或低峰期预先加载热点数据到缓存：
```python
# 批量加载热点商品
hot_products = get_hot_products_from_db()
for product in hot_products:
    redis.setex(f"product:{product.id}", 3600, json.dumps(product))
```

## 常见误区

### 1. 缓存越大越好
实际上，缓存容量需要根据实际的热点数据量来设置。过大的缓存不仅浪费内存，还可能导致淘汰策略效率低下。应该通过监控分析真正的热点数据分布。

### 2. 缓存永远不会失效
缓存必须设置合理的过期时间或失效策略。永不过期的缓存会导致数据陈旧，而且一旦缓存系统重启，所有数据都需要重新加载，造成数据库压力。

### 3. 缓存可以完全替代数据库
缓存是数据库的补充，不是替代品。缓存可能丢失（重启、内存不足），所以必须有数据库作为数据的最终来源。

### 4. 所有数据都应该缓存
缓存适合读多写少的热点数据。对于冷数据或者频繁变化的数据，缓存效果很差，甚至可能增加系统复杂性。

### 5. 缓存一致性很容易保证
在分布式系统中，缓存一致性是一个复杂的问题。简单的删除缓存策略在高并发下仍然可能出现不一致。需要根据业务场景选择合适的解决方案。

### 6. Redis 性能无限好
虽然 Redis 性能很好，但它仍然是单线程的（除了IO）。复杂命令（如 KEYS *、大KEY操作）会阻塞整个实例，影响其他请求。

### 7. 缓存穿透只影响性能
缓存穿透不仅影响性能，还可能被恶意利用进行DDoS攻击。必须通过布隆过滤器或空值缓存来防护。

### 8. TTL 设置越长越好
过长的TTL会导致数据陈旧，过短的TTL会增加数据库压力。应该根据数据的更新频率和业务容忍度来设置合理的TTL。

## 小结与思考

### 核心要点
- 缓存是提升系统性能的关键技术，可以显著降低延迟和成本
- 不同的缓存模式适用于不同的业务场景，需要根据一致性要求选择
- Redis 提供丰富的数据结构和功能，适合复杂的缓存需求
- 缓存问题（穿透、雪崩、击穿）需要针对性的解决方案
- 分布式环境下的缓存一致性是一个复杂但必须面对的问题
