# 图：万物互联的社交网络

上一节课我们学了树——分叉再分叉的层级结构。但现实世界远比树复杂：你的朋友圈不是一棵树，因为**你的朋友之间也互相认识**，形成了各种各样的环。今天我们要学的"图"，就是用来表示这种"随便连"的数据结构。

## 开场白

想象一下你打开微信，点开自己的朋友圈：你有50个好友，这50个人之间又互相加了好友，其中某些人还通过某个群加了你的同事……这可不是什么树形结构，因为**每个人都可以和任意人直接相连**，而且你可以通过多条路径找到同一个人。

再想想高德地图导航：从北京到上海，你可以走京沪高速，可以走京台高速，可以先飞到南京再高铁过去。每条路线都有不同的距离和时间，导航App要帮你找到"最优"的那条。

这两种场景有个共同点：**一堆东西，它们之间可以任意连接，连接还有"代价"**。在CS的世界里，我们把这种结构叫做——**图（Graph）**。

## 核心概念

### 什么是图？

图是最"自由"的数据结构——节点想连就连，想怎么连就怎么连。

简单说：**图就是一堆点（节点），点之间用线（边）连起来，爱怎么连怎么连**。

正式一点：图 G = (V, E)，其中 V 是**顶点集合**（vertices），E 是**边集合**（edges）。

```
   A ——— B ——— C
   |     |     |
   D ——— E     F
   |     |     |
   G ——— H ——— I ——— J
```

上面就是一个图。10个节点（A到J），若干条边把它们连起来。没有层级，没有根，想连就连。

### 图的类型

图不是一种东西，而是一"家子"。按照不同标准可以分成多种类型：

#### 按方向分：有向图 vs 无向图

| 类型 | 边的方向 | 生活类比 | 示例 |
|------|----------|----------|------|
| **无向图** | 边没有方向，A-B等于B-A | 微信好友（双向） | 朋友关系 |
| **有向图** | 边有方向，A→B不等于B→A | 微博关注（单向） | 关注/粉丝关系 |

```
无向图: A — B    表示A和B互相认识
有向图: A → B    表示A关注了B，但B不一定关注A
```

#### 按权重分：加权图 vs 无权图

| 类型 | 边的特征 | 生活类比 | 示例 |
|------|----------|----------|------|
| **无权图** | 边没有权重，只关心"连不连" | 是不是好友 | 社交网络连通性 |
| **加权图** | 边有权重（距离、代价、时间） | 好友的亲密度 | 地图上的公路距离 |

```
无权图: A — B     表示A和B相连
加权图: A —5— B   表示A和B相连，代价/距离是5
```

#### 连通性

- **连通图**：任意两个节点之间都有一条路径（至少能走到）
- **非连通图**：存在某些节点之间完全无法到达（图被"断开"了）
- **强连通图**（有向图专有）：任意两个节点之间可以互相到达

### 和树的对比

很多同学会问：**图不就是"不守规矩"的树吗？**

某种程度上，是的。树其实是图的一个"子集"——树是一种特殊的图，满足：
1. 连通（所有节点都能走到）
2. 无环（没有回路）
3. n个节点恰好n-1条边

图没有这些限制——可以有环、可以不连通、边数可以任意多。


**核心区别**：

| 特性 | BFS | DFS |
|------|-----|-----|
| 核心数据结构 | 队列（FIFO） | 栈（LIFO） |
| 搜索策略 | 一层一层往外扩 | 一条路走到底 |
| 最短路径 | 能找到无权图最短路径 | 不一定是最短 |
| 空间复杂度 | O(最宽层的宽度) | O(最大深度) |
| 典型应用 | 最短路径、社交距离 | 拓扑排序、迷宫求解 |

**生活化比喻**：
- **BFS**：你想知道"你和马斯克隔了几层关系"——先找直接好友，再找好友的好友，一层一层往外扩。
- **DFS**：你玩迷宫游戏——选一条路一直走，撞墙了就退回来换一条路。

在加权图中，我们常常需要找"最短路径"——不是步数最少，而是**总代价最小**。

```python
# code/examples/example-03-dijkstra.py
```

**运行结果**：

```
=== 城市之间的公路网络 ===
  北京 -> 天津(距离120), 济南(距离400)
  天津 -> 北京(距离120), 济南(距离300), 青岛(距离350)
  济南 -> 北京(距离400), 天津(距离300), 南京(距离600)
  南京 -> 济南(距离600), 青岛(距离450), 上海(距离300)

  青岛 -> 天津(距离350), 南京(距离450), 上海(距离700)

=== Dijkstra：从北京到所有城市的最短距离 ===
从北京出发：
  到 上海: 1170 公里
  到 南京: 920 公里
  到 天津: 120 公里
  到 济南: 400 公里
  到 青岛: 470 公里

=== 最短路径：北京 -> 上海 ===
最短路径: 北京 -> 天津 -> 青岛 -> 上海
总距离: 1170 公里

=== Dijkstra 注意事项 ===
1. Dijkstra不能处理负权边（负距离没意义，但负权重的图要用Bellman-Ford）
2. 如果所有边权重相同，Dijkstra退化成了BFS
3. Dijkstra是贪心算法：每次都选当前最近的，全局最优
```

**Dijkstra的核心思想**（用大白话说）：

1. 从起点出发，记录到所有其他节点的距离（初始为无穷大）
2. 每次都选"**目前已知距离最近的、还没处理过的节点**"
3. 用这个节点去"更新"它所有邻居的距离——如果从当前节点走更近，就更新
4. 重复2-3，直到所有节点都处理完

这就像你在北京，想知道去全国各地怎么走最近：先去最近的天津（120km），然后从天津出发，发现天津到青岛350km，所以北京→天津→青岛=470km，比北京直接到任何地方都合理。一步一步"扩散"出去。

## 深入理解

### 为什么BFS能找到最短路径？

在**无权图**中，BFS从起点开始，一层一层往外扩。第1层是直接邻居（距离1），第2层是邻居的邻居（距离2），以此类推。

**关键点**：BFS第一次遇到终点时，一定走的是最少层数的路径——因为如果存在更短的路径，BFS早就在更早的层遇到终点了。

这就是BFS天然适合找最短路径的原因：**层数 = 距离**。

### Dijkstra的贪心为什么是对的？

Dijkstra每次都选"当前最近的未处理节点"。为什么这种贪心策略能得到全局最优？

**直觉解释**：假设从起点S到节点X的最短路径经过中间节点M。如果S到M的最短路径已经确定了（距离d），那么从S经过M到X的距离就是d + w(M, X)。Dijkstra保证在选M的时候，已经找到了S到M的真正最短路径——因为如果存在更短的路径，那个路径上的某个未处理节点会比M更近，应该先被选。

这是一个**数学归纳法**式的论证——每一步的最优性保证了最终的全局最优。

### 时间/空间复杂度对比

| 操作 | 邻接矩阵 | 邻接表 |
|------|----------|--------|
| 添加边 | O(1) | O(1) |
| 删除边 | O(1) | O(邻居数) |
| 查询边 | O(1) | O(邻居数) |
| 遍历邻居 | O(n) | O(邻居数) |
| 空间 | O(n²) | O(n+m) |

| 算法 | 时间复杂度 | 空间复杂度 | 适用场景 |
|------|------------|------------|----------|
| BFS | O(V+E) | O(V) | 无权图最短路径、连通性 |
| DFS | O(V+E) | O(V) | 拓扑排序、连通分量 |
| Dijkstra | O((V+E)log V) | O(V) | 加权图最短路径 |

（V=节点数，E=边数）

### BFS/DFS在图和树上的区别

在树上遍历很简单——没有环，不会走回头路。但在图上遍历，**必须记录已访问的节点**，否则会在环里无限循环！

这就是为什么BFS和DFS都需要一个`visited`集合：

```python
visited = set()  # 记住你去过的地方
```

**没有visited的图遍历就像在一个环形走廊里跑步——永远跑不出去。**

## 实际应用

### 1. 社交网络：六度分隔理论

社交网络是图最经典的应用。每个人是一个节点，朋友关系是一条边。

Facebook研究发现：**地球上任意两个人之间，平均只隔3.57个中间人**。这就是著名的"六度分隔"理论——你和奥巴马之间，可能只隔着"你的朋友的朋友的...朋友"。

微信的"共同好友"、LinkedIn的"你可能认识的人"、抖音的"朋友的朋友"——底层都是图的遍历和邻接分析。

### 2. 地图导航：最短路径

高德地图、Google Maps的核心算法之一就是**Dijkstra及其变种A***算法。

每个路口是一个节点，每条路是一条边，权重是距离或时间。导航就是找从A到B的最短路径。

工业界用的是**A*算法**（Dijkstra的优化版，加入了启发式函数）和**Contraction Hierarchies**（层次化预处理），能在毫秒级算出全国范围内的最短路径。

### 3. 网络爬虫：图的遍历

搜索引擎的爬虫本质上就是**对互联网做一次大规模BFS**：

1. 从种子URL出发（比如google.com）
2. 解析页面，提取所有链接
3. 把新链接加入队列
4. 重复2-3，直到遍历完整个互联网

BFS保证爬虫先抓取"距离种子网站近"的重要页面，再逐渐深入。

### 4. 依赖管理：拓扑排序

pip install、npm install、Maven……这些包管理器在安装一个包之前，需要先安装它的依赖。

**依赖关系是一个有向无环图（DAG）**。包管理器用**拓扑排序**（基于DFS）来确定安装顺序。

```
你的项目
├── 依赖A
│   ├── 依赖C
│   └── 依赖D
└── 依赖B
    └── 依赖C  (C是共享依赖)
```

拓扑排序保证：**所有依赖都在被依赖之前安装**。

## 常见误区

⚠️ **注意这些坑！**

### 误区1：遍历图时忘记记录visited

这是新手最常犯的错误——在图上用BFS/DFS时，如果不记录哪些节点已经访问过，程序会在**环里无限循环**，直到爆栈或超时。

✅ **正确做法**：
```python
visited = set()
# BFS：加入队列时标记visited
# DFS：弹出/访问时标记visited
```

**关键细节**：BFS在"入队"时就标记visited，DFS在"出栈/访问"时标记。顺序不同，但目的都是防止重复处理。

### 误区2：混淆无权图和加权图的最短路径

- **无权图最短路径** → 用BFS（按层数算）
- **加权图最短路径** → 用Dijkstra（按总权重算）

用BFS去找加权图的最短路径是错的！因为BFS只看"经过几个节点"，不看"每条边有多长"。

```
BFS认为的最短: A → C → D  (2步)
加权实际最短:   A → B → D  (总权重更小)
```

✅ **正确做法**：看有没有权重。有权重用Dijkstra，没权重用BFS。如果所有权重相同，两者等价。

### 误区3：以为图必须是连通的

不！图可以有很多**连通分量**（connected components），每个分量内部的节点互相可达，但不同分量之间没有路径。

```
A — B    D — E — F
|   |    |
C —+     G

这是两个连通分量：{A, B, C} 和 {D, E, F, G}
```

✅ **正确做法**：如果要遍历整个非连通图，需要**对每个未访问的节点都启动一次BFS/DFS**。

### 练习1：判断图中是否有环 ⭐

**描述**：给定一个无向图的邻接表表示，判断图中是否存在环。

**要求**：
- 实现 `has_cycle(graph)` 函数
- graph是字典，key是节点，value是邻居列表
- 返回True（有环）或False（无环）

**提示**：在DFS遍历时，如果发现某个邻居已经访问过，而且这个邻居不是"来的方向"（父节点），说明有环。

<details>
<summary>💡 提示（点击展开）</summary>

DFS遍历时维护visited集合和parent信息。如果遇到一个已访问的节点，且它不是当前节点的父节点，则存在环。

```python
def has_cycle(graph):
    visited = set()
    for start in graph:
        if start not in visited:
            if _dfs_has_cycle(graph, start, visited, None):
                return True
    return False

def _dfs_has_cycle(graph, node, visited, parent):
    visited.add(node)
    for neighbor in graph.get(node, []):
        if neighbor not in visited:
            if _dfs_has_cycle(graph, neighbor, visited, node):
                return True
        elif neighbor != parent:
            return True  # 发现回边，有环！
    return False
```

### 练习2：计算社交网络中的"好友距离" ⭐⭐

**描述**：给定一个社交网络（无向图），计算从某个人出发，到其他所有人的"好友距离"（最短需要经过几个人）。

**要求**：
- 实现 `friend_distances(graph, person)` 函数
- 返回一个字典，key是人名，value是距离（自己是0）
- 不可达的人标记为-1

**提示**：这是BFS的直接应用。BFS的"层数"就是好友距离。

<details>
<summary>💡 提示（点击展开）</summary>

用BFS遍历，在队列中记录(当前人, 距离)。每扩散一层，距离+1。

```python
from collections import deque

def friend_distances(graph, person):
    if person not in graph:
        return {}
    distances = {p: -1 for p in graph}
    distances[person] = 0
    queue = deque([(person, 0)])
    while queue:
        current, dist = queue.popleft()
        for friend in graph.get(current, []):
            if distances[friend] == -1:
                distances[friend] = dist + 1
                queue.append((friend, dist + 1))
    return distances
```

</details>

### 练习3：实现拓扑排序 ⭐⭐⭐

**描述**：给定一个有向无环图（DAG），返回它的拓扑排序——一个节点的线性序列，使得对于每条有向边u→v，u在序列中都出现在v之前。

**要求**：
- 实现 `topological_sort(graph)` 函数
- graph是字典，key是课程，value是这门课的前置课程列表
- 返回一个合法的学习顺序，如果图中有环则返回None

**提示**：可以用DFS（后序遍历的逆序）或Kahn算法（不断删除入度为0的节点）。

<details>
<summary>💡 提示（点击展开）</summary>

方法1（DFS）：对每个未访问的节点做DFS，完成后将节点加入结果列表的头部。

```python
def topological_sort(graph):
    visited = set()
    in_stack = set()  # 检测环用
    result = []

    def dfs(node):
        if node in in_stack:
            return False  # 有环
        if node in visited:
            return True
        in_stack.add(node)
        for prereq in graph.get(node, []):
            if not dfs(prereq):
                return False
        in_stack.remove(node)
        visited.add(node)
        result.append(node)
        return True

    for course in graph:
        if course not in visited:
            if not dfs(course):
                return None  # 有环，无法拓扑排序
    return result
```

</details>

## 小结与思考

### 本节要点

1. **图是最通用的数据结构**：节点任意连接，可以有环、可以不连通、边可以有权重
2. **两种表示方法**：邻接矩阵（查边快、占空间）和邻接表（省空间、遍历快）
3. **BFS和DFS**：BFS按层扩散（用队列），DFS深入优先（用栈），遍历时必须记录visited
4. **Dijkstra算法**：加权图最短路径，贪心策略+优先队列，不能处理负权边
5. **图的应用无处不在**：社交网络、地图导航、网络爬虫、依赖管理

### 思考题

1. **如果微信要做一个"可能认识的人"推荐功能，你会怎么设计？** 提示：可以考虑"共同好友数量"、"好友的好友"、BFS的层数等。如果用户有1亿人，怎么高效地计算推荐？

2. **Dijkstra不能处理负权边，那负权边到底有什么问题？** 想想如果有一条边的权重是-5，Dijkstra的贪心策略会出什么错？（提示：Dijkstra认为"选过的节点就是最优的"，但负权边可能让已经"选过"的路径变得更优。）

3. **有向无环图（DAG）和树有什么区别？** 一个节点在DAG中可以有多个"父节点"（比如课程A的前置是B和C，而B和C没有先后关系），这在树中是不允许的。想想Git的提交历史——为什么它是一个DAG而不是树？
