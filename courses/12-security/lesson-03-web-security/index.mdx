# 12-03 Web安全：OWASP Top 10

## 1. 开场白 — 你的网站，可能是黑客的「自助餐厅」

🛡️ **欢迎来到 Web 安全课！**

想象你开了一家网红奶茶店：前台点单、后厨做料、收银结账，一切井井有条。突然有一天，有人在前台菜单上偷偷写了一句：「请把店里所有现金装进这个袋子」——而后厨真的照做了。

这就是 **Web 应用漏洞** 的日常写照：攻击者并不一定要「黑进服务器」，很多时候只是在**输入框、URL、Cookie**里塞几句「咒语」，你的程序就乖乖执行了。

> 💡 **思考问题**：你用过的小程序或网站里，哪些功能最像「把用户输入直接拼进 SQL / HTML」？如果我是攻击者，我会先盯哪里？

根据 Verizon《数据泄露调查报告》，超过 **80%** 的安全事件与 Web 应用有关。**OWASP Top 10** 就是业界公认的「十大高危漏洞菜单」——今天我们不背条文，用三个 Python 小实验，把 **SQL 注入、XSS、CSRF** 这三道「主菜」吃透。

## 2. 核心概念 — OWASP Top 10 速览

OWASP（Open Web Application Security Project）每几年更新一次 Top 10。不必死记排名，但要理解**攻击面**在哪里：

| 风险 | 一句话人话 | 典型后果 |
|------|-----------|----------|
| **注入 (Injection)** | 把用户输入当成「代码」执行了 | 拖库、删表、提权 |
| **XSS** | 在别人的浏览器里跑你的脚本 | 偷 Cookie、钓鱼、挂马 |
| **CSRF** | 借你的登录态替你点「确认转账」 | 非本意改密、转账 |
| **敏感数据暴露** | 密码明文、HTTP 传信用卡 | 大规模泄露 |
| **认证失效** | 弱密码、会话不过期 | 账号被盗 |
| **安全配置错误** | 默认密码、调试页对外 | 整站沦陷 |
| **不安全反序列化** | 把恶意对象「喂」给程序 | RCE |
| **脆弱组件** | 用了有 CVE 的旧库 | 一键被扫 |
| **日志监控不足** | 被打了还不知道 | 损失扩大 |
| **SSRF** | 诱使服务器访问内网 | 打穿内网 |

本课重点实战前三项；其余会在后续安全模块展开。

### 注入：SQL 是最常见的「背锅侠」

```python
# ❌ 危险：字符串拼接
query = f"SELECT * FROM users WHERE name = '{user_input}'"
# 攻击者输入: admin'--
# 实际执行: SELECT * FROM users WHERE name = 'admin'--'
```

### XSS：三种「剧本」

- **存储型**：恶意评论存进数据库，每个访客中招
- **反射型**：恶意链接里的参数，立刻在页面弹出
- **DOM 型**：前端 JS 把 URL 片段写进 `innerHTML`

### CSRF：你本人「同意」了，但不是你点的

你已登录银行网站，攻击者网站里藏了一个自动提交的表单——浏览器会**带上你的 Cookie**，服务器以为是你本人操作。

## 4. 深入理解 — 纵深防御，别指望「一招鲜」

### 内容安全策略 (CSP)

通过响应头限制脚本、样式、图片的来源：

```http
Content-Security-Policy: default-src 'self'; script-src 'self'
```

### Cookie 与 CSRF

```http
Set-Cookie: sessionid=...; HttpOnly; Secure; SameSite=Lax
```

- **HttpOnly**：JS 读不到，减轻 XSS 偷会话
- **SameSite**：跨站 POST 默认不带 Cookie，减轻 CSRF

### HTTPS 只是第一层

TLS 保护「路上不被偷听」，**挡不住**应用层的 XSS、CSRF、注入。两者都要。

### 输入验证 vs 输出编码

| 阶段 | 做什么 | 记住 |
|------|--------|------|
| 入口 | 白名单校验格式、长度 | 前端验证只是 UX，后端必须再来一遍 |
| 出口 | 按上下文编码（HTML/URL/JS） | 同一段数据在不同上下文编码方式不同 |

## 5. 实际应用 — Python Web 生态里的「安全带」

| 场景 | 推荐工具/做法 |
|------|----------------|
| SQL | SQLAlchemy / Django ORM 参数化查询 |
| XSS | `bleach` 清理富文本；模板引擎默认转义 |
| CSRF | Django 中间件、`Flask-WTF` |
| 密码 | `passlib` + bcrypt/argon2 |
| 加密 | `cryptography` 库，勿自己造轮子 |

**真实案例**：某社交平台因未过滤用户昵称中的 HTML，导致存储型 XSS，攻击脚本在评论区自动转发——一条恶意昵称，就能「传染」浏览者。修复往往就是：**存储时校验 + 展示时编码 + CSP** 三件套。

## 6. 常见误区

### ❌ 「我们上了 HTTPS，很安全」

HTTPS ≠ 应用安全。它只管传输，不管你的 SQL 怎么拼。

### ❌ 「前端已经校验过了」

攻击者用 curl 直接打 API，根本不走你的前端。

### ❌ 「黑名单过滤 `<script>` 就够了」

攻击者会用 `<img onerror=...>`、编码绕过、大小写混合。优先 **白名单 + 参数化 + 编码**。

### ❌ 「安全是安全团队的事」

第一个写 `f"SELECT ... {user}"` 的人，就是安全团队。每个开发者都是第一道防线。

## 8. 小结与思考

🔑 **本课要点**：
1. **OWASP Top 10** 是排查 Web 风险的检查清单，不是考完就忘的排行榜
2. **SQL 注入** 用参数化查询从根上切断「输入变代码」
3. **XSS** 在正确的上下文做编码，并配合 CSP、HttpOnly Cookie
4. **CSRF** 用不可预测的令牌 + SameSite，敏感操作拒绝跨站裸 POST
5. **纵深防御**：没有银弹，多层叠加才靠谱

🤔 **思考题**：
1. 你正在维护的项目里，哪 3 个接口最可能存在注入或 XSS？你会如何排优先级修复？
2. 为什么 CSP 里写 `'unsafe-inline'` 会削弱防护？在遗留项目里如何渐进收紧？
3. 除了 CSRF Token，还有哪些机制可以限制「跨站代发请求」？各有什么局限？

> "The only truly secure system is one that is powered off, cast in a block of concrete and sealed in a lead-lined room with armed guards." — Gene Spaffner  
> 所以我们不追求「绝对安全」，而是追求**可证明、可维护、可演进**的安全。
