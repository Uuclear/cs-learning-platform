# 挑战2：日志分析与监控系统

## 背景
你是一家互联网公司的后端工程师，负责构建一个日志分析系统。系统需要处理大量的应用日志，提取关键指标并生成监控报告。

## 数据模型
```java
class LogEntry {
    String requestId;           // 请求ID
    String serviceName;         // 服务名称
    String methodName;          // 方法名称
    long responseTime;          // 响应时间（毫秒）
    int statusCode;             // HTTP状态码
    LocalDateTime timestamp;    // 时间戳
    String errorMessage;        // 错误信息（可能为null）
    String userId;              // 用户ID（可能为null）
}
```

## 任务要求

### 基础任务（必须完成）
1. **性能监控**：计算每个服务的平均响应时间、95分位响应时间和慢请求比例（>1000ms）
2. **错误分析**：按服务和错误类型统计错误率，并找出错误最多的前5个方法
3. **流量分析**：按小时统计每小时的请求数量，识别流量高峰时段

### 进阶任务（选做）
4. **异常检测**：实现一个异常检测算法，自动识别响应时间或错误率异常的服务
5. **用户行为分析**：分析用户的请求模式，识别异常用户（如请求频率过高、错误率过高等）
6. **关联分析**：找出经常同时出现错误的服务组合（可能表示依赖关系问题）

## 约束条件
- 必须使用 Stream API 的所有高级特性，包括分组、分区、归约等
- 需要处理大规模数据（假设每分钟有数万条日志）
- 代码应该具有良好的扩展性，便于添加新的分析维度
- 考虑内存使用效率，避免不必要的中间集合创建

## 提示
- 使用 `Collectors.partitioningBy()` 进行二分统计（如成功/失败）
- 使用自定义收集器处理复杂的聚合逻辑
- 考虑使用 `DoubleSummaryStatistics` 获取完整的统计信息
- 对于分位数计算，可以先排序再取指定位置的值
- 使用 `LocalDateTime` 的各种方法进行时间分组

## 验证标准
- 能够正确处理各种边界情况（空日志、无效数据等）
- 性能优化良好，能够实时处理大量日志
- 代码结构模块化，不同的分析功能相互独立
- 充分展示了 Stream API 的强大功能

## 扩展思考
- 如何将这个系统改造为流式处理系统（处理实时日志流）？
- 如果日志数据存储在分布式文件系统中，应该如何设计处理流程？
- 如何实现动态的监控规则配置（而不是硬编码在代码中）？
- 如何保证分析结果的准确性（考虑数据丢失、重复等问题）？
