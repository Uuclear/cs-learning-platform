# 通信协议：I2C、SPI与UART

## 开场白：三种语言对话的比喻

想象一下，你是一位国际外交官，需要与来自不同国家的代表进行沟通。有的代表只会说一种特定的语言，有的可以用多种方式交流，而你的任务就是找到最适合的沟通方式。

在单片机的世界里，我们面临着同样的挑战！不同的外设芯片就像是来自不同国家的代表，它们有着各自的"语言习惯"——这就是我们今天要学习的三种主要通信协议：**UART**、**I2C** 和 **SPI**。

- **UART** 就像是两个人用对讲机通话——简单直接，一次只能一个人说话
- **I2C** 就像是在一个会议室里，所有人都听着主持人点名发言——节约线路但速度较慢  
- **SPI** 就像是高速专线电话——全双工通信，速度快但需要更多线路

选择合适的"语言"对于成功建立通信至关重要。今天，我们就来深入学习这三种协议的奥秘！

## 核心概念：三种协议详解

### UART（通用异步收发传输器）

UART 是最古老也是最简单的串行通信协议之一。它的核心特点包括：

- **异步通信**：不需要共享时钟信号，双方通过预设的波特率同步
- **点对点通信**：通常只连接两个设备（TX 发送，RX 接收）
- **简单线路**：只需要两根线（加上地线共三根）
- **波特率**：表示每秒传输的比特数，常见的有 9600、115200 等

UART 的数据帧格式通常包含：
- 1位起始位（低电平）
- 5-9位数据位
- 可选的奇偶校验位
- 1-2位停止位（高电平）

### I2C（Inter-Integrated Circuit，两线制串行总线）

I2C 是由 Philips（现在的 NXP）开发的同步串行通信协议，最大的特点是**多主多从**架构：

- **两线制**：SCL（时钟线）+ SDA（数据线）
- **地址机制**：每个从设备都有唯一的7位或10位地址
- **开漏输出**：需要上拉电阻，支持多设备共享总线
- **主从模式**：主设备发起通信，从设备响应

I2C 的通信过程：
1. 主设备发送起始条件（SCL高电平时SDA从高变低）
2. 发送从设备地址 + 读写位
3. 从设备确认（ACK）
4. 传输数据字节
5. 每个字节后都有ACK/NACK
6. 主设备发送停止条件（SCL高电平时SDA从低变高）

### SPI（Serial Peripheral Interface，串行外设接口）

SPI 是 Motorola 开发的高速同步串行接口，特点是**全双工**通信：

- **四线制**：SCLK（时钟）、MOSI（主出从入）、MISO（主入从出）、CS/SS（片选）
- **高速传输**：没有地址概念，直接点对点通信
- **全双工**：可以同时发送和接收数据
- **灵活配置**：支持不同的时钟极性和相位（CPOL/CPHA）

SPI 的工作原理：
- 主设备通过拉低对应的 CS 线选择从设备
- 主设备产生时钟信号 SCLK
- 数据在 MOSI 和 MISO 线上同时传输
- 通信结束后主设备拉高 CS 线

## 深入理解：协议对比与选型指南

| 特性 | UART | I2C | SPI |
|------|------|-----|-----|
| **线路数量** | 2-3根 | 2根 | 3-4根 + CS |
| **通信速度** | 慢（通常<1Mbps） | 中等（标准100kbps，快速400kbps） | 快（可达几十Mbps） |
| **设备数量** | 点对点 | 多从设备（理论上127个） | 多从设备（每个需要独立CS） |
| **同步方式** | 异步（波特率同步） | 同步（共享时钟） | 同步（共享时钟） |
| **全双工** | 半双工 | 半双工 | 全双工 |
| **复杂度** | 简单 | 中等 | 中等 |
| **距离** | 较长（可达几十米） | 短（通常<1米） | 短（通常<0.5米） |

### 选型建议：

- **选择 UART 当**：需要简单点对点通信，如调试输出、蓝牙模块、GPS模块
- **选择 I2C 当**：引脚资源紧张，需要连接多个传感器，如温湿度传感器、加速度计
- **选择 SPI 当**：需要高速数据传输，如显示屏、SD卡、高速ADC/DAC

## 实际应用：真实世界中的协议使用

### OLED 屏幕（SPI）

OLED 显示屏通常使用 SPI 接口，因为需要高速刷新显示内容。典型的连接方式：
- SCLK → 时钟线
- MOSI → 数据线  
- DC → 数据/命令选择
- CS → 片选
- RST → 复位

### 温湿度传感器（I2C）

像 DHT20、SHT30 这样的温湿度传感器通常使用 I2C，因为：
- 节省引脚（只需要两根线）
- 传感器本身不需要高速通信
- 多个传感器可以共享同一总线

### 蓝牙模块（UART）

HC-05、ESP32 等蓝牙/WiFi 模块通常通过 UART 与主控通信，因为：
- 协议简单，易于实现
- 可以长距离连接（通过串口转USB）
- 调试方便，可以直接用串口监视器查看

## 常见误区：避坑指南

### I2C 地址冲突

**问题**：多个 I2C 设备使用相同的地址导致通信失败

**解决方案**：
- 查看设备手册确认地址
- 使用 I2C 扫描工具检测总线上的设备
- 选择可配置地址的设备版本
- 使用 I2C 多路复用器

### 波特率不匹配

**问题**：UART 通信时双方波特率设置不一致导致乱码

**解决方案**：
- 确保双方使用完全相同的波特率
- 注意晶振精度对波特率的影响
- 使用标准波特率（9600, 19200, 115200等）

### SPI 时序问题

**问题**：不同设备对 CPOL/CPHA 设置要求不同

**解决方案**：
- 仔细阅读设备数据手册
- 理解四种 SPI 模式（0,1,2,3）
- 使用逻辑分析仪验证时序

### 上拉电阻缺失

**问题**：I2C 总线缺少上拉电阻导致通信不稳定

**解决方案**：
- I2C 总线必须有上拉电阻（通常4.7kΩ）
- 多个设备时可能需要调整阻值
- 避免上拉电阻过大或过小

## 小结与思考

今天我们学习了单片机通信的三大协议：

- **UART**：简单可靠的异步通信，适合点对点连接
- **I2C**：节约引脚的同步总线，适合多传感器系统  
- **SPI**：高速全双工接口，适合大数据量传输

**思考题**：
1. 如果你需要连接10个相同的温度传感器，应该选择哪种协议？为什么？
2. 在电池供电的物联网设备中，如何平衡通信速度和功耗？
3. 当 SPI 和 I2C 都能满足需求时，你会如何做出选择？

记住，没有最好的协议，只有最适合的协议。理解每种协议的特点和适用场景，才能在实际项目中做出正确的技术选择。
