8:T6ea6,<h1>14-06 敏捷开发：Scrum与看板</h1>
<h2>开场白</h2>
<p>大家好！欢迎来到敏捷开发的世界！🚀</p>
<p>在当今快速变化的软件行业中，传统的瀑布式开发方法已经难以满足市场对速度和灵活性的需求。想象一下：你的竞争对手每两周就能发布新功能，而你的团队还在为一个六个月前规划的功能苦苦挣扎——这正是许多传统开发团队面临的困境。</p>
<p>敏捷开发应运而生，它不仅仅是一套方法论，更是一种思维方式的转变。从僵化的计划驱动到灵活的价值驱动，从文档导向到可工作的软件导向，敏捷让我们能够更快地响应变化、更好地满足用户需求。</p>
<p>在这个课程中，我们将深入探索两种最流行的敏捷框架：Scrum和看板。你将学习如何组织高效的敏捷团队、如何进行冲刺规划、如何使用燃尽图跟踪进度，以及如何通过看板方法实现持续交付。无论你是开发者、项目经理还是产品负责人，这些知识都将帮助你在实际工作中更好地实践敏捷。</p>
<h2>核心概念</h2>
<h3>敏捷宣言与原则</h3>
<p>2001年，17位软件开发专家在美国犹他州雪鸟滑雪胜地聚会，共同起草了《敏捷软件开发宣言》。这份宣言确立了四个核心价值观：</p>
<ol>
<li><strong>个体和互动</strong> 高于 流程和工具</li>
<li><strong>可工作的软件</strong> 高于 详尽的文档  </li>
<li><strong>客户合作</strong> 高于 合同谈判</li>
<li><strong>响应变化</strong> 高于 遵循计划</li>
</ol>
<p>这些价值观背后是12条敏捷原则，强调早期和持续交付有价值的软件、欢迎需求变更、频繁交付可工作的软件等。</p>
<h3>Scrum框架</h3>
<p>Scrum是最流行的敏捷框架之一，它通过固定的迭代周期（冲刺）来组织工作：</p>
<p><strong>三个核心角色：</strong></p>
<ul>
<li><strong>产品负责人</strong>（Product Owner）：负责最大化产品价值，管理产品待办列表</li>
<li><strong>Scrum Master</strong>：确保Scrum流程被正确执行，移除团队障碍</li>
<li><strong>开发团队</strong>：跨职能的自组织团队，负责交付可工作的软件</li>
</ul>
<p><strong>四个关键仪式：</strong></p>
<ul>
<li><strong>冲刺计划会议</strong>（Sprint Planning）：选择本次冲刺要完成的工作</li>
<li><strong>每日站会</strong>（Daily Scrum）：15分钟同步进展和计划</li>
<li><strong>冲刺评审会议</strong>（Sprint Review）：展示完成的工作并获取反馈</li>
<li><strong>冲刺回顾会议</strong>（Sprint Retrospective）：反思流程并制定改进计划</li>
</ul>
<p><strong>三个主要工件：</strong></p>
<ul>
<li><strong>产品待办列表</strong>（Product Backlog）：所有已知需求的优先级列表</li>
<li><strong>冲刺待办列表</strong>（Sprint Backlog）：本次冲刺承诺完成的工作</li>
<li><strong>增量</strong>（Increment）：冲刺结束时可用的软件版本</li>
</ul>
<h3>用户故事与验收标准</h3>
<p>用户故事是敏捷中描述需求的标准格式：</p>
<blockquote>
<p><strong>作为</strong> [用户角色]，<strong>我想要</strong> [功能]，<strong>以便</strong> [商业价值]</p>
</blockquote>
<p>每个用户故事都应该有明确的验收标准，定义"完成"的标准。例如：</p>
<ul>
<li>功能必须通过所有自动化测试</li>
<li>必须有相应的文档更新</li>
<li>必须通过用户体验审核</li>
</ul>
<h3>故事点与速度估算</h3>
<p>故事点是一种相对估算单位，用于评估用户故事的复杂度、工作量和风险。团队通过计划扑克等技术进行估算，避免精确的时间估算。</p>
<p><strong>团队速度</strong>（Velocity）是指团队在一个冲刺中能够完成的故事点数，用于预测未来的交付能力。</p>
<h3>看板方法</h3>
<p>看板起源于丰田生产系统，强调可视化工作流和限制在制品数量（WIP）：</p>
<p><strong>核心实践：</strong></p>
<ul>
<li><strong>可视化工作流</strong>：使用看板板显示任务状态（如待办、进行中、已完成）</li>
<li><strong>限制WIP</strong>：设置每个状态列的最大任务数量，减少上下文切换</li>
<li><strong>管理流动</strong>：关注任务在整个系统中的流动效率</li>
<li><strong>明确流程规则</strong>：定义何时可以移动任务到下一状态</li>
<li><strong>实施反馈循环</strong>：定期回顾和改进流程</li>
</ul>
<p><strong>关键指标：</strong></p>
<ul>
<li><strong>周期时间</strong>（Cycle Time）：任务从开始到完成的时间</li>
<li><strong>吞吐量</strong>（Throughput）：单位时间内完成的任务数量</li>
<li><strong>累积流图</strong>（Cumulative Flow Diagram）：显示各状态任务数量随时间的变化</li>
</ul>

# -*- coding: utf-8 -*-
"""
示例1：冲刺计划模拟
模拟冲刺规划过程：计算团队速度，根据优先级和容量选择用户故事
"""

from typing import List, Dict, Tuple


def calculate_team_velocity(completed_story_points: List[int]) -&gt; float:
    """
    计算团队速度（平均完成的故事点数）

    :param completed_story_points: 过去几个冲刺中完成的故事点数列表
    :return: 团队平均速度
    """
    if not completed_story_points:
        return 0.0
    return sum(completed_story_points) / len(completed_story_points)


def select_user_stories_for_sprint(
    backlog: List[Dict],
    team_velocity: float,
    buffer_ratio: float = 0.2
) -&gt; Tuple[List[Dict], float]:
    """
    根据团队速度和优先级选择冲刺中的用户故事

    :param backlog: 用户故事待办列表，每个故事包含 'id', 'title', 'story_points', 'priority'
    :param team_velocity: 团队速度（可完成的故事点数）
    :param buffer_ratio: 缓冲比例，用于应对不确定性（默认20%）
    :return: (选中的用户故事列表, 实际使用的故事点数)
    """
    # 计算实际可用容量（考虑缓冲）
    available_capacity = team_velocity * (1 - buffer_ratio)

    # 按优先级排序（优先级数字越小越重要）
    sorted_backlog = sorted(backlog, key=lambda story: story['priority'])

    selected_stories = []
    total_points = 0.0

    for story in sorted_backlog:
        # 检查是否还有足够容量
        if total_points + story['story_points'] &lt;= available_capacity:
            selected_stories.append(story)
            total_points += story['story_points']
        else:
            # 如果当前故事太大，跳过（在实际中可能会拆分故事）
            continue

    return selected_stories, total_points


def main():
    """主函数：演示冲刺规划过程"""
    # 模拟过去3个冲刺的完成情况
    past_velocity = [25, 30, 28]  # 故事点数
    current_velocity = calculate_team_velocity(past_velocity)

    print(f"团队历史速度: {past_velocity}")
    print(f"计算出的团队速度: {current_velocity:.1f} 故事点")

    # 模拟产品待办列表
    product_backlog = [
        {'id': 'US-001', 'title': '用户登录功能', 'story_points': 8, 'priority': 1},
        {'id': 'US-002', 'title': '密码重置功能', 'story_points': 5, 'priority': 2},
        {'id': 'US-003', 'title': '用户个人资料页面', 'story_points': 13, 'priority': 3},
        {'id': 'US-004', 'title': '搜索功能优化', 'story_points': 8, 'priority': 4},
        {'id': 'US-005', 'title': '通知系统', 'story_points': 21, 'priority': 5},
        {'id': 'US-006', 'title': '数据导出功能', 'story_points': 3, 'priority': 6},
    ]

    print("\n产品待办列表:")
    for story in product_backlog:
        print(f"  {story['id']}: {story['title']} ({story['story_points']}点, 优先级{story['priority']})")

    # 选择冲刺故事
    selected_stories, used_points = select_user_stories_for_sprint(
        product_backlog, current_velocity
    )

    print(f"\n冲刺容量: {current_velocity * 0.8:.1f} 故事点 (考虑20%缓冲)")
    print(f"选中的用户故事 ({used_points:.1f} 故事点):")
    for story in selected_stories:
        print(f"  ✓ {story['id']}: {story['title']} ({story['story_points']}点)")

    remaining_capacity = current_velocity * 0.8 - used_points
    print(f"\n剩余容量: {remaining_capacity:.1f} 故事点")


if __name__ == "__main__":
    main()

"""
示例2：燃尽图数据生成
从冲刺进度生成ASCII燃尽图数据（每日剩余故事点数 vs 理想线）
"""

from typing import List, Tuple
import math


def calculate_burndown_data(
    total_story_points: int,
    daily_progress: List[int],
    sprint_days: int = 10
) -&gt; Tuple[List[int], List[float]]:
    """
    计算燃尽图数据

    :param total_story_points: 冲刺总故事点数
    :param daily_progress: 每日完成的故事点数列表
    :param sprint_days: 冲刺天数（默认10天）
    :return: (实际剩余点数列表, 理想剩余点数列表)
    """
    # 确保daily_progress长度不超过sprint_days
    if len(daily_progress) &gt; sprint_days:
        daily_progress = daily_progress[:sprint_days]

    # 补齐到sprint_days长度（如果数据不足）
    while len(daily_progress) &lt; sprint_days:
        daily_progress.append(0)

    # 计算实际剩余故事点数
    actual_remaining = [total_story_points]
    cumulative_done = 0

    for day_progress in daily_progress:
        cumulative_done += day_progress
        remaining = max(0, total_story_points - cumulative_done)
        actual_remaining.append(remaining)

    # 计算理想燃尽线（线性下降）
    ideal_remaining = []
    for day in range(sprint_days + 1):
        ideal_points = total_story_points * (1 - day / sprint_days)
        ideal_remaining.append(ideal_points)

    return actual_remaining, ideal_remaining


def generate_ascii_burndown_chart(
    actual_remaining: List[int],
    ideal_remaining: List[float],
    chart_width: int = 50,
    chart_height: int = 20
) -&gt; str:
    """
    生成ASCII燃尽图

    :param actual_remaining: 实际剩余故事点数列表
    :param ideal_remaining: 理想剩余故事点数列表
    :param chart_width: 图表宽度
    :param chart_height: 图表高度
    :return: ASCII图表字符串
    """
    if not actual_remaining:
        return "没有数据可显示"

    max_points = max(max(actual_remaining), max(ideal_remaining))
    if max_points == 0:
        max_points = 1

    # 创建图表网格
    grid = [[' ' for _ in range(chart_width)] for _ in range(chart_height)]

    # 绘制坐标轴
    for i in range(chart_height):
        grid[i][0] = '|'
    for j in range(chart_width):
        grid[chart_height-1][j] = '-'
    grid[chart_height-1][0] = '+'

    # 绘制理想线
    for day, points in enumerate(ideal_remaining):
        x = int(day * (chart_width - 2) / (len(ideal_remaining) - 1)) + 1
        y = chart_height - 2 - int(points * (chart_height - 2) / max_points)
        if 0 &lt;= y &lt; chart_height - 1 and 0 &lt;= x &lt; chart_width:
            grid[y][x] = '.'

    # 绘制实际线
    for day, points in enumerate(actual_remaining):
        x = int(day * (chart_width - 2) / (len(actual_remaining) - 1)) + 1
        y = chart_height - 2 - int(points * (chart_height - 2) / max_points)
        if 0 &lt;= y &lt; chart_height - 1 and 0 &lt;= x &lt; chart_width:
            grid[y][x] = '*'

    # 生成图表字符串
    chart_lines = []
    for row in grid:
        chart_lines.append(''.join(row))

    # 添加图例
    chart_lines.append("")
    chart_lines.append("图例: * = 实际进度, . = 理想进度")
    chart_lines.append(f"Y轴: 剩余故事点数 (0-{int(max_points)})")
    chart_lines.append(f"X轴: 冲刺天数 (0-{len(actual_remaining)-1})")

    return '\n'.join(chart_lines)


def main():
    """主函数：演示燃尽图生成"""
    # 示例数据
    total_points = 40
    daily_done = [5, 8, 6, 4, 7, 3, 2, 1, 0, 0]  # 前10天的完成情况

    print("冲刺燃尽图演示")
    print("=" * 50)
    print(f"冲刺总故事点数: {total_points}")
    print(f"每日完成点数: {daily_done}")

    actual_remaining, ideal_remaining = calculate_burndown_data(
        total_points, daily_done, sprint_days=10
    )

    print(f"\n实际剩余点数: {actual_remaining}")
    print(f"理想剩余点数: {[round(x, 1) for x in ideal_remaining]}")

    # 生成并显示ASCII图表
    chart = generate_ascii_burndown_chart(actual_remaining, ideal_remaining)
    print("\n燃尽图:")
    print(chart)


if __name__ == "__main__":
    main()

# -*- coding: utf-8 -*-
"""
示例3：看板指标计算
从任务完成数据计算看板指标：周期时间、吞吐量、在制品数量（WIP）
"""

from typing import List, Dict, Tuple
from datetime import datetime, timedelta


def calculate_cycle_time(
    tasks: List[Dict]
) -&gt; Tuple[float, List[int]]:
    """
    计算周期时间（任务从开始到完成的时间）

    :param tasks: 任务列表，每个任务包含 'start_date', 'end_date'
    :return: (平均周期时间, 所有任务的周期时间列表)
    """
    cycle_times = []

    for task in tasks:
        if task['start_date'] and task['end_date']:
            start = datetime.fromisoformat(task['start_date'])
            end = datetime.fromisoformat(task['end_date'])
            cycle_time_days = (end - start).days
            cycle_times.append(cycle_time_days)

    avg_cycle_time = sum(cycle_times) / len(cycle_times) if cycle_times else 0
    return avg_cycle_time, cycle_times


def calculate_throughput(
    tasks: List[Dict],
    period_start: str,
    period_end: str
) -&gt; float:
    """
    计算吞吐量（在指定时间段内完成的任务数量）

    :param tasks: 任务列表
    :param period_start: 统计周期开始日期 (ISO格式)
    :param period_end: 统计周期结束日期 (ISO格式)
    :return: 平均每日吞吐量
    """
    period_start_dt = datetime.fromisoformat(period_start)
    period_end_dt = datetime.fromisoformat(period_end)
    period_days = (period_end_dt - period_start_dt).days

    if period_days &lt;= 0:
        return 0.0

    completed_tasks = 0
    for task in tasks:
        if task['end_date']:
            end_dt = datetime.fromisoformat(task['end_date'])
            if period_start_dt &lt;= end_dt &lt;= period_end_dt:
                completed_tasks += 1

    return completed_tasks / period_days


def calculate_wip(
    tasks: List[Dict],
    date_point: str
) -&gt; int:
    """
    计算指定日期的在制品数量（WIP - Work In Progress）

    :param tasks: 任务列表，每个任务包含 'start_date', 'end_date'
    :param date_point: 查询日期 (ISO格式)
    :return: 在制品数量
    """
    query_date = datetime.fromisoformat(date_point)
    wip_count = 0

    for task in tasks:
        start_dt = datetime.fromisoformat(task['start_date']) if task['start_date'] else None
        end_dt = datetime.fromisoformat(task['end_date']) if task['end_date'] else None

        # 任务在查询日期处于进行中状态
        if start_dt and start_dt &lt;= query_date and (not end_dt or end_dt &gt; query_date):
            wip_count += 1

    return wip_count


def calculate_cumulative_flow(
    tasks: List[Dict],
    start_date: str,
    end_date: str
) -&gt; Dict[str, List[int]]:
    """
    计算累积流图数据

    :param tasks: 任务列表
    :param start_date: 开始日期
    :param end_date: 结束日期
    :return: 包含各状态任务数量的字典
    """
    start_dt = datetime.fromisoformat(start_date)
    end_dt = datetime.fromisoformat(end_date)

    # 生成日期范围
    current = start_dt
    dates = []
    while current &lt;= end_dt:
        dates.append(current.isoformat()[:10])  # YYYY-MM-DD
        current += timedelta(days=1)

    # 初始化累积数据
    cumulative_data = {
        'todo': [],
        'in_progress': [],
        'done': []
    }

    for date_str in dates:
        date_dt = datetime.fromisoformat(date_str)

        todo_count = 0
        in_progress_count = 0
        done_count = 0

        for task in tasks:
            start_dt_task = datetime.fromisoformat(task['start_date']) if task['start_date'] else None
            end_dt_task = datetime.fromisoformat(task['end_date']) if task['end_date'] else None

            if end_dt_task and end_dt_task &lt;= date_dt:
                done_count += 1
            elif start_dt_task and start_dt_task &lt;= date_dt:
                in_progress_count += 1
            else:
                todo_count += 1

        cumulative_data['todo'].append(todo_count)
        cumulative_data['in_progress'].append(in_progress_count)
        cumulative_data['done'].append(done_count)

    return cumulative_data

<h2>深入理解</h2>
<h3>估算的艺术</h3>
<p>故事点估算不是关于时间，而是关于相对复杂度。团队使用斐波那契数列（1, 2, 3, 5, 8, 13...）或T恤尺码（XS, S, M, L, XL）来进行估算，因为随着复杂度增加，估算的不确定性也呈指数增长。</p>
<p><strong>计划扑克</strong>（Planning Poker）是一种有效的估算技术：</p>
<ol>
<li>每个团队成员独立选择一个估算值</li>
<li>同时展示选择</li>
<li>讨论差异最大的估算值</li>
<li>重复直到达成共识</li>
</ol>
<h3>燃尽图的解读</h3>
<p>燃尽图不仅仅是进度跟踪工具，更是问题发现的雷达：</p>
<ul>
<li><strong>实际线高于理想线</strong>：进度落后，可能需要调整范围或寻求帮助</li>
<li><strong>实际线低于理想线</strong>：进度超前，可能有额外容量处理技术债务</li>
<li><strong>平坦的线段</strong>：可能存在阻塞问题或估算不准确</li>
<li><strong>突然下降</strong>：可能有大的用户故事被完成，或者估算过于粗糙</li>
</ul>
<h3>看板的流动效率</h3>
<p>看板的核心是优化流动效率（Flow Efficiency），即增值时间占总周期时间的比例。典型的软件开发团队流动效率只有5-15%，大部分时间花在等待、审批、上下文切换等非增值活动上。</p>
<p>通过限制WIP和可视化瓶颈，团队可以显著提高流动效率：</p>
<ul>
<li>减少同时进行的任务数量</li>
<li>识别并解决流程瓶颈</li>
<li>建立明确的完成定义（Definition of Done）</li>
<li>实施持续集成和自动化测试</li>
</ul>
<h3>混合方法</h3>
<p>在实践中，很多团队采用ScrumBan（Scrum + Kanban）的混合方法：</p>
<ul>
<li>保留Scrum的固定迭代和角色结构</li>
<li>使用看板的可视化和WIP限制来管理冲刺内的工作流</li>
<li>根据团队成熟度和项目特点灵活调整</li>
</ul>
<p>这种方法结合了Scrum的结构化和看板的灵活性，特别适合维护和支持工作较多的团队。</p>
<h2>实际应用</h2>
<h3>工具选择指南</h3>
<p><strong>Jira</strong>：功能最全面的企业级工具，适合大型团队和复杂项目</p>
<ul>
<li>优势：强大的报告、集成、权限控制</li>
<li>劣势：学习曲线陡峭，成本较高</li>
</ul>
<p><strong>Trello</strong>：简单直观的看板工具，适合小型团队和个人项目</p>
<ul>
<li>优势：易用性好，免费版功能充足</li>
<li>劣势：缺乏高级报告和复杂工作流</li>
</ul>
<p><strong>Azure DevOps</strong>：微软生态系统的集成解决方案</p>
<ul>
<li>优势：与Visual Studio和GitHub深度集成</li>
<li>劣势：主要面向微软技术栈</li>
</ul>
<p><strong>ClickUp</strong>：全能型项目管理工具</p>
<ul>
<li>优势：功能丰富，价格合理</li>
<li>劣势：界面复杂，可能功能过剩</li>
</ul>
<h3>团队规模考虑</h3>
<ul>
<li><strong>小型团队</strong>（3-7人）：Scrum效果最佳，沟通成本低</li>
<li><strong>中型团队</strong>（8-15人）：考虑拆分为多个Scrum团队或使用看板</li>
<li><strong>大型团队</strong>（15+人）：需要规模化敏捷框架（如SAFe、LeSS）</li>
</ul>
<h3>远程团队最佳实践</h3>
<ol>
<li><strong>异步沟通优先</strong>：减少会议依赖，使用文档和工具同步信息</li>
<li><strong>可视化工作流</strong>：确保所有人都能清楚看到工作状态</li>
<li><strong>定期同步</strong>：保持团队连接感，但控制会议时间</li>
<li><strong>明确的约定</strong>：定义清晰的工作协议和响应时间期望</li>
</ol>
<h3>度量与改进</h3>
<p>建立持续改进的循环：</p>
<ol>
<li><strong>收集数据</strong>：速度、周期时间、缺陷率、客户满意度</li>
<li><strong>分析趋势</strong>：关注长期趋势而非单点数据</li>
<li><strong>实验改进</strong>：基于数据提出假设并实验验证</li>
<li><strong>标准化成功</strong>：将有效的改进措施制度化</li>
</ol>
<h2>常见误区</h2>
<h3>❌ 敏捷就是没有文档</h3>
<p><strong>事实</strong>：敏捷强调"可工作的软件高于详尽的文档"，但并不意味着不需要文档。关键是创建<strong>有价值的文档</strong>，如架构决策记录、API文档、用户手册等，而不是为了文档而文档。</p>
<h3>❌ Scrum Master就是项目经理</h3>
<p><strong>事实</strong>：Scrum Master是服务型领导，专注于移除障碍和促进流程，而不是分配任务或管理进度。真正的项目经理职责在Scrum中被分散到产品负责人（范围管理）和开发团队（进度管理）。</p>
<h3>❌ 故事点可以直接转换为小时</h3>
<p><strong>事实</strong>：故事点是相对估算单位，不同团队的故事点含义不同。试图将故事点转换为小时会破坏估算的相对性和团队的自主性。</p>
<h3>❌ 每日站会就是状态汇报</h3>
<p><strong>事实</strong>：每日站会是团队的同步和协调会议，重点是识别障碍和协调工作，而不是向经理汇报进度。理想情况下，经理不应该参加每日站会。</p>
<h3>❌ 敏捷适合所有项目</h3>
<p><strong>事实</strong>：敏捷最适合需求不确定、变化频繁的项目。对于需求明确、技术成熟、法规严格的项目（如医疗设备、航空航天），传统的预测性方法可能更合适。</p>
<h3>❌ 可以只采用部分敏捷实践</h3>
<p><strong>事实</strong>：敏捷是一个整体系统，单独采用某些实践（如每日站会但不进行迭代交付）往往会导致"伪敏捷"，无法获得真正的收益。</p>
