## 核心概念

### 什么是总线？

**总线**就是计算机内部传输数据的"公共通道"。

用最通俗的话说：
- **CPU** = 城市CBD（中央商务区），处理各种事务
- **内存** = 仓库，临时存放货物（数据）
- **硬盘** = 大型物流园，长期存储
- **显卡** = 印刷厂，处理图像输出
- **总线** = 连接这些地方的**公路网**

**为什么叫"总线"？** "总"的意思就是"公共的"——所有设备共享同一条通道。就像一条城市主干道，所有车辆（数据）都要在这条路上跑。

### 总线为什么重要？

早期的计算机，每个部件之间都拉一根线直连。结果就是——**线多到爆炸**。

```
CPU ---- 内存    (2根线)
CPU ---- 硬盘    (2根线)
CPU ---- 显卡    (2根线)
CPU ---- 键盘    (2根线)
CPU ---- 鼠标    (2根线)
... 以此类推，几十个设备就是几十对线
```

后来聪明的工程师们一拍大腿：**为什么不让大家共用一条路呢？**

于是总线诞生了！所有设备挂到同一根"总线"上，谁需要通信谁就用。大大减少了线路数量。

### 总线的三驾马车：地址线、数据线、控制线

一条完整的总线由三种"线路"组成，各司其职：

| 线路类型 | 功能 | 类比 |
|---------|------|------|
| **地址线**（Address Bus） | 告诉数据"要去哪里" | 快递单上的"收件地址" |
| **数据线**（Data Bus） | 传输实际的数据 | 快递包裹本身 |
| **控制线**（Control Bus） | 发送控制信号（读/写/中断等） | 快递员的"签收确认"流程 |

举个栗子 🌰：CPU想从内存读取地址 `0x1000` 的数据：

```
第1步：CPU通过地址线发送 "0x1000" → 告诉内存"我要这个地址的数据"
第2步：CPU通过控制线发送 "读"信号    → 告诉内存"是读取，不是写入"
第3步：内存通过数据线返回数据        → 把数据顺着数据线送回CPU
```

**注意**：地址线通常是**单向**的（CPU→内存），而数据线是**双向**的（可读可写）。

让我们通过代码来理解这些概念：

### 示例1: 总线通信模拟

```python
# 总线通信模拟
# 演示地址线、数据线、控制线如何协同工作

class BusLine:
    """模拟一根总线线路"""
    def __init__(self, name, direction):
        self.name = name          # 线路名称
        self.direction = direction # 方向：单向或双向
        self.data = None          # 当前传输的数据

    def send(self, data):
        """发送数据"""
        self.data = data
        print(f"  [{self.name}] 发送: {data}")

    def receive(self):
        """接收数据"""
        return self.data


class SimpleBus:
    """
    简化的总线模拟器
    包含地址线、数据线和控制线
    """
    def __init__(self, address_bits=32, data_bits=64):
        # 地址线：32位，单向（CPU→内存）
        self.address_bus = BusLine("地址线(32位)", "单向: CPU→内存")
        # 数据线：64位，双向
        self.data_bus = BusLine("数据线(64位)", "双向")
        # 控制线：发送读/写信号
        self.control_bus = BusLine("控制线", "双向")

    def read_memory(self, cpu, memory, address):
        """CPU从内存读取数据（模拟总线操作）"""
        print(f"\n📖 读操作: CPU从内存地址 {hex(address)} 读取数据")
        print("-" * 50)

        # 第1步：地址线传输地址
        self.address_bus.send(hex(address))

        # 第2步：控制线发送"读"信号
        self.control_bus.send("READ")

        # 第3步：内存读取数据，通过数据线返回
        data = memory.read(address)
        self.data_bus.send(data)

        # 第4步：CPU接收数据
        cpu.receive(data)
        print(f"  ✅ CPU收到数据: {data}")

    def write_memory(self, cpu, memory, address, data):
        """CPU向内存写入数据（模拟总线操作）"""
        print(f"\n✏️ 写操作: CPU向内存地址 {hex(address)} 写入 {data}")
        print("-" * 50)

        # 第1步：地址线传输地址
        self.address_bus.send(hex(address))

        # 第2步：数据线传输要写入的数据
        self.data_bus.send(data)

        # 第3步：控制线发送"写"信号
        self.control_bus.send("WRITE")

        # 第4步：内存接收数据
        memory.write(address, data)
        print(f"  ✅ 数据已写入内存")


class SimpleMemory:
    """简易内存"""
    def __init__(self):
        self.data = {}

    def read(self, address):
        return self.data.get(address, 0)

    def write(self, address, value):
        self.data[address] = value

    def dump(self):
        print("  当前内存内容:")
        for addr, val in sorted(self.data.items()):
            print(f"    {hex(addr)}: {val}")


class SimpleCPU:
    """简易CPU"""
    def __init__(self):
        self.register = None  # 寄存器

    def receive(self, data):
        self.register = data


def main():
    """主程序：演示总线读写操作"""
    print("=== 总线通信模拟 ===\n")

    # 创建组件
    bus = SimpleBus()
    cpu = SimpleCPU()
    memory = SimpleMemory()

    # 先写入一些数据
    bus.write_memory(cpu, memory, 0x1000, 42)
    bus.write_memory(cpu, memory, 0x1004, 256)
    bus.write_memory(cpu, memory, 0x1008, 1024)

    memory.dump()

    # 再从内存读取数据
    bus.read_memory(cpu, memory, 0x1000)
    print(f"  CPU寄存器当前值: {cpu.register}")

    bus.read_memory(cpu, memory, 0x1008)
    print(f"  CPU寄存器当前值: {cpu.register}")


if __name__ == "__main__":
    main()

# 输出:
# === 总线通信模拟 ===
#
# ✏️ 写操作: CPU向内存地址 0x1000 写入 42
# --------------------------------------------------
#   [地址线(32位)] 发送: 0x1000
#   [数据线(64位)] 发送: 42
#   [控制线] 发送: WRITE
#   ✅ 数据已写入内存
#
# ... 更多写操作 ...
#
#   当前内存内容:
#     0x1000: 42
#     0x1004: 256
#     0x1008: 1024
#
# 📖 读操作: CPU从内存地址 0x1000 读取数据
# --------------------------------------------------
#   [地址线(32位)] 发送: 0x1000
#   [控制线] 发送: READ
#   [数据线(64位)] 发送: 42
#   ✅ CPU收到数据: 42
```

**代码解释**：
- 三根总线各司其职：地址线告诉"去哪"，数据线负责"搬运"，控制线指挥"读还是写"
- 每次总线操作都是一个"协议"：先送地址，再送控制信号，最后传输数据
- 真实硬件中这些操作在纳秒级别完成，CPU的每个指令都离不开总线

### 示例2: 总线带宽计算

```python
# 总线带宽计算演示
# 帮助理解频率、位宽和带宽的关系

def calculate_bandwidth(frequency_mhz, bus_width_bits, cycles_per_transfer=1):
    """
    计算总线带宽
    公式: 带宽 = 频率 × 位宽 / 8 / 每传输周期数
    单位: MB/s (兆字节每秒)

    参数:
        frequency_mhz: 总线频率（MHz）
        bus_width_bits: 总线位宽（位）
        cycles_per_transfer: 每次传输需要的周期数（默认1）
    """
    # 频率转换为每秒传输次数
    transfers_per_second = frequency_mhz * 1_000_000 / cycles_per_transfer

    # 每次传输的字节数
    bytes_per_transfer = bus_width_bits / 8

    # 带宽 = 传输次数 × 每次字节数
    bandwidth_bytes = transfers_per_second * bytes_per_transfer
    bandwidth_mbps = bandwidth_bytes / (1024 * 1024)  # 转换为MB/s
    bandwidth_gbps = bandwidth_mbps / 1024            # 转换为GB/s

    return bandwidth_mbps, bandwidth_gbps


def compare_bus_bandwidth():
    """对比各种常见总线的理论带宽"""
    print("=== 总线带宽对比 ===\n")

    # 定义各种总线：(名称, 频率MHz, 位宽, 每传输周期数)
    buses = [
        ("PCIe 3.0 x1",     8000,   1,   1),    # 8GT/s，1位有效
        ("PCIe 3.0 x16",    8000,   16,  1),    # 16条lane
        ("PCIe 4.0 x16",    16000,  16,  1),    # 16GT/s，翻倍
        ("PCIe 5.0 x16",    32000,  16,  1),    # 32GT/s，再翻倍
        ("DDR4-3200",       3200,   64,  2),    # DDR=双数据传输
        ("DDR5-6000",       6000,   64,  2),    # 更快
        ("USB 3.0",         5000,   1,   1),    # 5Gbps
        ("USB 3.2 Gen2x2",  10000,  2,   1),    # 双通道
        ("SATA 3.0",        6000,   1,   1),    # 6Gbps
    ]

    print(f"{'总线名称':<20} {'频率(MHz)':>12} {'位宽':>6} {'带宽(MB/s)':>14} {'带宽(GB/s)':>12}")
    print("-" * 70)

    for name, freq, width, cycles in buses:
        mbps, gbps = calculate_bandwidth(freq, width, cycles)
        print(f"{name:<20} {freq:>12,} {width:>6} {mbps:>14,.0f} {gbps:>12,.2f}")

    print("\n💡 关键发现:")
    print("  1. 带宽 = 频率 × 位宽，两者都很重要")
    print("  2. PCIe x16是x1的16倍速，所以显卡都要用x16插槽")
    print("  3. DDR的'Double Data Rate'让每个周期传2次数据，带宽翻倍")
    print("  4. 新一代PCIe每次迭代带宽翻倍！")


def bandwidth_bottle_neck_demo():
    """演示带宽瓶颈问题"""
    print("\n\n=== 带宽瓶颈演示 ===\n")

    # 假设场景：从SSD加载一个5GB的游戏到内存
    game_size_gb = 5
    game_size_mb = game_size_gb * 1024

    interfaces = [
        ("SATA 3.0 SSD", 600),       # ~600 MB/s 实际
        ("NVMe PCIe 3.0", 3500),     # ~3500 MB/s
        ("NVMe PCIe 4.0", 7000),     # ~7000 MB/s
        ("NVMe PCIe 5.0", 14000),    # ~14000 MB/s
    ]

    print(f"加载 {game_size_gb}GB 游戏所需时间:\n")
    for name, speed in interfaces:
        time_sec = game_size_mb / speed
        if time_sec < 1:
            time_str = f"{time_sec * 1000:.0f}毫秒"
        else:
            time_str = f"{time_sec:.2f}秒"
        bar = "█" * int(time_sec * 2)
        print(f"  {name:<20} {speed:>6} MB/s → {time_str:>10} {bar}")

    print("\n💡 带宽就像公路的车道数：车道越多，同时通过的车越多！")


if __name__ == "__main__":
    compare_bus_bandwidth()
    bandwidth_bottle_neck_demo()

# 输出:

#
# 总线名称                 频率(MHz)     位宽     带宽(MB/s)    带宽(GB/s)
# ----------------------------------------------------------------------
# PCIe 3.0 x1                8,000      1            977         0.95
# PCIe 3.0 x16               8,000     16         15,625        15.26
# PCIe 4.0 x16              16,000     16         31,250        30.52
# PCIe 5.0 x16              32,000     16         62,500        61.04
# DDR4-3200                  3,200     64         25,000        24.41
# ...
#
# 💡 关键发现:
#   1. 带宽 = 频率 × 位宽，两者都很重要
#   2. PCIe x16是x1的16倍速，所以显卡都要用x16插槽
#   ...
```

**代码解释**：
- 总线带宽由两个因素决定：频率（每秒传多少次）和位宽（每次传多少位）
- 带宽不够就会产生"瓶颈"——CPU再快，数据送不过来也白搭
- 这就是为什么"好马配好鞍"：高端CPU+低端主板=性能浪费

### 示例3: 总线仲裁算法

```python
# 总线仲裁算法演示
# 当多个设备同时想使用总线时，谁先用？

import time


class BusArbiter:
    """
    总线仲裁器
    决定哪个设备获得总线使用权
    """
    def __init__(self, algorithm="round_robin"):
        self.algorithm = algorithm    # 仲裁算法
        self.current_index = 0        # 轮询起始位置
        self.busy = False             # 总线是否被占用
        self.current_master = None    # 当前总线主人

    def grant_access(self, devices):
        """
        授予总线使用权
        支持三种算法：轮询、优先级、固定优先级
        """
        if self.busy:
            print(f"  ⏳ 总线正被 {self.current_master} 占用，等待...")
            return None

        if self.algorithm == "round_robin":
            return self._round_robin(devices)
        elif self.algorithm == "priority":
            return self._priority(devices)
        elif self.algorithm == "fixed_priority":
            return self._fixed_priority(devices)
        return None

    def _round_robin(self, devices):
        """轮询算法：公平轮转，一人一次"""
        if not devices:
            return None

        # 从当前位置开始找下一个请求者
        for i in range(len(devices)):
            idx = (self.current_index + i) % len(devices)
            if devices[idx]["requesting"]:
                self.current_index = (idx + 1) % len(devices)
                winner = devices[idx]
                self.busy = True
                self.current_master = winner["name"]
                return winner
        return None

    def _priority(self, devices):
        """动态优先级：高优先级先用，用完降优先级"""
        # 按优先级排序，选最高的请求者
        requesting = [d for d in devices if d["requesting"]]
        if not requesting:
            return None
        winner = max(requesting, key=lambda d: d["priority"])
        self.busy = True
        self.current_master = winner["name"]
        # 降低获胜者优先级（让其他设备也有机会）
        winner["priority"] = max(1, winner["priority"] - 1)
        return winner

    def _fixed_priority(self, devices):
        """固定优先级：CPU永远第一，依次递减"""
        for device in sorted(devices, key=lambda d: d["fixed_priority"], reverse=True):
            if device["requesting"]:
                self.busy = True
                self.current_master = device["name"]
                return device
        return None

    def release(self):
        """释放总线"""
        self.busy = False
        print(f"  🔓 {self.current_master} 释放了总线")
        self.current_master = None


def simulate_arbitration():
    """模拟总线仲裁过程"""
    print("=== 总线仲裁模拟 ===\n")

    # 定义三个设备
    devices = [
        {"name": "CPU",         "requesting": True,  "priority": 10, "fixed_priority": 3},
        {"name": "DMA控制器",    "requesting": True,  "priority": 7,  "fixed_priority": 2},
        {"name": "显卡GPU",     "requesting": True,  "priority": 5,  "fixed_priority": 1},
    ]

    # 场景1: 轮询算法
    print("场景1: 轮询仲裁（Round Robin）")
    print("-" * 40)
    arbiter = BusArbiter("round_robin")

    for round_num in range(1, 6):
        print(f"\n第 {round_num} 轮:")
        winner = arbiter.grant_access(devices)
        if winner:
            print(f"  🏆 {winner['name']} 获得总线使用权")
            # 模拟传输
            time.sleep(0.01)  # 稍微暂停
            arbiter.release()
            # 每轮每个设备都请求
            for d in devices:
                d["requesting"] = True

    # 场景2: 优先级算法
    print("\n\n场景2: 动态优先级仲裁")
    print("-" * 40)

    # 重置设备
    devices = [
        {"name": "CPU",         "requesting": True,  "priority": 10, "fixed_priority": 3},
        {"name": "DMA控制器",    "requesting": True,  "priority": 7,  "fixed_priority": 2},
        {"name": "显卡GPU",     "requesting": True,  "priority": 5,  "fixed_priority": 1},
    ]

    arbiter2 = BusArbiter("priority")

    for round_num in range(1, 6):
        print(f"\n第 {round_num} 轮 (优先级: CPU={devices[0]['priority']}, DMA={devices[1]['priority']}, GPU={devices[2]['priority']}):")
        winner = arbiter2.grant_access(devices)
        if winner:
            print(f"  🏆 {winner['name']} 获得总线使用权 (优先级: {winner['priority']+1})")
            arbiter2.release()

    # 场景3: 对比总结
    print("\n\n📊 仲裁算法对比:")
    print("-" * 40)
    print("  算法          | 公平性 | 实时性 | 复杂度")
    print("  " + "-" * 35)
    print("  轮询          | ★★★★★  | ★★★    | 低")
    print("  动态优先级     | ★★★★   | ★★★★   | 中")
    print("  固定优先级     | ★★     | ★★★★★  | 低")
    print("\n💡 实际系统中，CPU通常有最高优先级，")
    print("   但也会用超时机制防止其他设备饿死。")


if __name__ == "__main__":
    simulate_arbitration()

# 输出:
# === 总线仲裁模拟 ===
#
# 场景1: 轮询仲裁（Round Robin）
# ----------------------------------------
#
# 第 1 轮:
#   🏆 CPU 获得总线使用权
#   🔓 CPU 释放了总线
#
# 第 2 轮:
#   🏆 DMA控制器 获得总线使用权
#   🔓 DMA控制器 释放了总线
#
# 第 3 轮:
#   🏆 显卡GPU 获得总线使用权
#   🔓 显卡GPU 释放了总线
# ...
```

**代码解释**：
- 当多个设备同时想用总线时，需要"仲裁"来决定谁先用
- 轮询算法最公平，一人一轮
- 优先级算法让重要设备先用，但会降低优先级防止"饿死"其他设备
- 真实硬件中仲裁在几个时钟周期内完成，快到不可思议

## 深入理解

### 总线的类型：分层的高速公路系统

计算机内部的总线不是"一根线"，而是一个**分层网络**，就像城市的道路系统：

```
                    CPU
                     │
              ┌──────┼──────┐
              │      │      │
         前端总线  内存总线  内部总线
         (FSB)    (Memory)  (Internal)
              │      │      │
         北桥芯片   内存    CPU缓存
              │
         ┌────┼────┐
         │         │
      PCIe总线   I/O总线
         │         │
      显卡/SSD    USB/SATA
```

| 总线类型 | 连接对象 | 速度 | 类比 |
|---------|---------|------|------|
| **内部总线** | CPU内部寄存器、ALU、缓存 | 最快 | 厨房内部的传送带 |
| **前端总线（FSB）** | CPU ↔ 北桥芯片 | 很快 | 高速公路主干道 |
| **内存总线** | CPU/北桥 ↔ 内存 | 非常快 | 专用货运通道 |
| **PCIe总线** | 北桥 ↔ 显卡/SSD等 | 快 | 城市快速路 |
| **I/O总线（USB/SATA）** | 南桥 ↔ 外设 | 中等 | 城市普通道路 |

> 现代CPU已经把内存控制器和PCIe控制器集成到CPU内部了，北桥芯片"下岗"了。但概念上还是一样的分层结构。

### 总线时序：红绿灯和交通规则

总线通信不是"想说就说"的，必须遵守严格的时序：

```
时钟周期:  T1    T2    T3    T4    T5    T6
          ┌────┬────┬────┬────┬────┬────┐
地址线:    │ADDR│    │    │    │    │    │  ← T1: 放地址
控制线:    │    │READ│    │    │    │    │  ← T2: 发读信号
数据线:    │    │    │    │DATA│    │    │  ← T4: 数据到位
          └────┴────┴────┴────┴────┴────┘
                     ↑ 等待内存响应（wait states）
```

**关键概念**：
- **时钟周期**：总线的"心跳"，每个周期做一次操作
- **等待周期（Wait States）**：如果内存慢于CPU，CPU要等几个周期才能收到数据
- **同步 vs 异步**：同步总线有统一时钟，异步总线靠握手信号

### 总线带宽计算：核心公式

```
带宽 = 频率 × 位宽 / 8
```

举个通俗的例子：
- 频率 = 每秒发车次数（1000辆/秒）
- 位宽 = 每辆车的载货量（64箱）
- 带宽 = 每秒运货量 = 1000 × 64 / 8 = 8000 箱/秒

**为什么DDR能翻倍？** DDR（Double Data Rate）在时钟的"上升沿"和"下降沿"各传一次数据，相当于同样频率下运货次数翻倍。

### 总线 vs 点对点：架构演进

```
传统总线架构:                    现代点对点架构:
    ┌─────┐                          ┌─────┐
    │ CPU │←── 共享总线 ──→设备       │ CPU │──直连──→内存
    └─────┘                          └──┬──┘
   问题：设备多了会"堵车"                 ├──直连──→PCIe设备
                                       └──直连──→PCIe设备
                                      优势：各走各的，互不干扰
```

现代计算机已经从"共享总线"演进到"点对点直连"，但"总线"这个概念和术语一直沿用下来。

## 实际应用

### 应用场景1: 装机选主板——PCIe版本很关键

你花1万块买了个RTX 4090显卡，结果插在了PCIe 3.0的主板上？

**后果**：显卡性能损失约5-15%，具体取决于负载。

**原因**：PCIe 3.0 x16带宽约16 GB/s，PCIe 4.0约32 GB/s，PCIe 5.0约64 GB/s。高端显卡在高负载时确实能跑满带宽。

**建议**：
- 买主板时注意：PCIe版本要与显卡匹配
- PCIe是向下兼容的，但高速设备插低速口=浪费性能
- 显卡插最靠近CPU的那个x16插槽（通常是直连CPU的）

### 应用场景2: SSD速度选择——SATA vs NVMe

| 类型 | 接口 | 理论带宽 | 实际速度 | 适合场景 |
|------|------|---------|---------|---------|
| SATA SSD | SATA 3.0 | 6 Gbps | ~550 MB/s | 日常办公、仓库盘 |
| NVMe PCIe 3.0 | M.2 PCIe 3.0 x4 | 32 Gbps | ~3500 MB/s | 游戏加载、一般创作 |
| NVMe PCIe 4.0 | M.2 PCIe 4.0 x4 | 64 Gbps | ~7000 MB/s | 视频编辑、大型项目 |
| NVMe PCIe 5.0 | M.2 PCIe 5.0 x4 | 128 Gbps | ~14000 MB/s | 发烧级、未来proof |

加载一个50GB的3A大作：
- SATA SSD：约90秒
- PCIe 3.0 NVMe：约14秒
- PCIe 4.0 NVMe：约7秒

差距巨大！总线带宽直接决定了你的等待时间。

## 常见误区

⚠️ **注意这些坑！**

### 误区1: "总线频率越高越好"

**错误理解**：
> 这个主板总线频率1600MHz，那个只有800MHz，肯定选高的！

**真相**：
带宽 = 频率 × 位宽。只看频率不看位宽是不对的。一个800MHz × 64位的总线（6400 MB/s）比1600MHz × 32位的总线（6400 MB/s）带宽一样！

✅ **正确做法**：
- 看带宽而不是只看频率
- 注意位宽：x1、x4、x8、x16差距巨大
- 实际应用中，延迟往往比带宽更影响体验

### 误区2: "所有插槽都是一样的"

**错误理解**：
> 主板上有三个PCIe插槽，随便插哪个都行吧？

**真相**：
- 最上面的x16插槽通常直连CPU，延迟最低、带宽最高
- 下面的PCIe插槽可能走南桥芯片（PCH），要"转一道手"
- 有些x16物理长度的插槽实际只有x4电气带宽

✅ **正确做法**：
- 显卡永远插最上面的x16插槽
- 高速NVMe SSD优先插CPU直连的M.2插槽
- 网卡、声卡等低速设备可以走南桥

### 误区3: "USB都是同一个速度"

**错误理解**：
> 都是USB接口，插哪个不一样？

**真相**：
USB的名称和速度堪称计算机界的"命名灾难"：

| 名称 | 曾用名 | 最大速度 |
|------|--------|---------|
| USB 2.0 | - | 480 Mbps |
| USB 3.0 | USB 3.1 Gen 1 / USB 3.2 Gen 1 | 5 Gbps |
| USB 3.1 | USB 3.1 Gen 2 / USB 3.2 Gen 2 | 10 Gbps |
| USB 3.2 | USB 3.2 Gen 2x2 | 20 Gbps |
| USB4 | - | 40 Gbps |

✅ **正确做法**：
- 传大文件认准USB 3.0以上（蓝色接口通常是3.0）
- USB Type-C ≠ 高速，Type-C只是接口形状
- 需要高速传输时查一下主板规格表

## 动手实践

现在轮到你动手了！

### 练习1: 简单总线带宽计算器 ⭐

**描述**：
编写一个带宽计算器，根据总线的频率和位宽计算理论带宽。

**要求**：
- 实现 `calculate_bandwidth(frequency_mhz, bus_width_bits, ddr=False)` 函数
- 支持DDR模式（带宽翻倍）
- 返回结果以 MB/s 和 Gbps 两种单位显示
- 格式化输出，保留两位小数

**示例**：
```python
# DDR4-3200内存总线: 1600MHz, 64位, DDR模式
bw = calculate_bandwidth(1600, 64, ddr=True)
print(f"带宽: {bw['mbps']:.2f} MB/s, {bw['gbps']:.2f} Gbps")
# 输出: 带宽: 25600.00 MB/s, 204.80 Gbps
```

<details>
<summary>💡 提示（点击展开）</summary>

- 基本公式：带宽 = 频率 × 位宽 / 8
- DDR模式下结果乘以2
- MB/s 转换为 Gbps：1 MB/s = 8 Mbps，再除以1000得到Gbps
- 注意单位换算：1 MB = 10^6 字节（带宽通常用十进制）

</details>

### 练习2: 多设备总线仲裁模拟器 ⭐⭐

**描述**：
实现一个总线仲裁模拟器，支持多种仲裁算法，并统计每个设备的总线使用率。

**要求**：
- 实现 `BusArbiter` 类，支持 `round_robin` 和 `priority` 两种算法
- 多个设备可以同时请求总线
- 每轮仲裁后统计每个设备的"获取次数"和"等待次数"
- 模拟100轮，输出统计数据

**示例场景**：
```python
devices = [
    {"name": "CPU",     "priority": 10},
    {"name": "DMA",     "priority": 7},
    {"name": "GPU",     "priority": 5},
    {"name": "网卡",    "priority": 3},
]

# 模拟每个设备随机请求（可以设置请求概率）
arbiter = BusArbiter("round_robin")
arbiter.simulate(devices, rounds=100)
arbiter.print_stats()
# 输出每个设备获得了多少次总线使用权
```

<details>
<summary>💡 提示（点击展开）</summary>

- 用随机数模拟设备是否发出请求（例如CPU 90%概率请求，网卡10%概率）
- 统计每个设备的 `granted_count` 和 `wait_count`
- 轮询算法用循环指针实现，记录上次分配位置
- 优先级算法每次选优先级最高的请求者

</details>

### 练习3: 总线拓扑与延迟模拟 ⭐⭐⭐

**描述**：
模拟不同总线拓扑下的数据传输延迟，对比共享总线和点对点架构的性能差异。

**要求**：
- 实现两种拓扑：`SharedBusTopology` 和 `PointToPointTopology`
- 每个设备有到"中心节点"（北桥/交换机）的延迟
- 数据从设备A到设备B需要经过中心节点
- 共享总线：多个设备同时传输时需要排队（串行）
- 点对点：不同设备对可以同时传输（并行）
- 模拟100次随机传输，计算平均延迟和总时间

**对比指标**：
- 平均延迟（单次传输）
- 总完成时间（100次传输的总耗时）
- 总线利用率（共享总线被占用的时间比例）

<details>
<summary>💡 提示（点击展开）</summary>

- 共享拓扑：每次传输时间 = 排队等待 + 传输时间，同时请求要串行
- 点对点：不同设备对的传输可以并行，只有相同设备对才需要排队
- 模拟时随机选择源设备和目标设备
- 可以用简单的"时间片"模型，每个时间片处理一个或多个传输

</details>

## 小结与思考

### 本节要点

1. **总线是计算机的"高速公路"** —— 连接CPU、内存、I/O设备的公共通道，没有总线各部件就是一盘散沙

2. **总线三要素** —— 地址线（去哪）、数据线（传什么）、控制线（怎么传），三者缺一不可

3. **带宽公式** —— 带宽 = 频率 × 位宽，两者同等重要，不能只看一个

4. **分层总线架构** —— 从内部总线到前端总线、内存总线、PCIe、USB，不同层次速度不同、用途不同

5. **仲裁机制** —— 多个设备争抢总线时，通过轮询或优先级决定谁先用，公平和效率要兼顾

6. **从共享到点对点** —— 现代计算机已从共享总线演进为点对点直连，但"总线"概念仍在

### 思考题

1. **为什么现代CPU要集成内存控制器？** 从总线的延迟和带宽角度分析，把内存控制器放在北桥芯片里 vs 放在CPU内部，各有什么优劣？

2. **如果让你设计下一代总线架构，你会如何平衡带宽、延迟、成本三者之间的关系？** 有没有可能让所有设备都点对点直连？为什么？

3. **USB4支持40Gbps，但你的USB设备可能只有5Gbps，这会不会造成浪费？** 从总线兼容性和未来演进的角度思考，高速总线向下兼容低速设备的意义是什么？
