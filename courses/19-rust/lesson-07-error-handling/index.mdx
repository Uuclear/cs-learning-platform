# 错误处理：Result与panic

## 开场白：编程世界的天气预报

想象一下，你是一名经验丰富的飞行员，准备执行一次重要的飞行任务。在起飞前，你会查看天气预报，对吧？但天气预报从来不会说"今天绝对晴朗"或"今天绝对有暴风雨"，而是告诉你"有70%的概率会下雨"。

Rust 的错误处理哲学就像这种理性的天气预报——它不假设一切都会顺利，也不假设一切都会失败，而是**明确地告诉你哪些事情可能会出错，以及如何优雅地应对这些情况**。

在其他编程语言中，错误处理往往像是在飞行途中突然遇到暴风雨——程序崩溃，用户困惑，开发者头疼。但在 Rust 中，错误处理是**编译时就强制考虑的**，就像起飞前就必须制定好应急预案一样。

今天，我们就来学习 Rust 如何通过 `Result<T, E>` 类型、`?` 操作符和 `panic!` 宏，帮助我们编写更加健壮和可靠的代码。

## 核心概念：Result、panic、? 操作符

### Result<T, E> 类型：可恢复错误的优雅解决方案

Rust 使用 `Result<T, E>` 枚举来处理**可恢复的错误**。它的定义很简单：

```rust
enum Result<T, E> {
    Ok(T),
    Err(E),
}
```

- `Ok(T)` 表示操作成功，包含成功的结果值
- `Err(E)` 表示操作失败，包含错误信息

这种设计的美妙之处在于：**你无法忽略错误的可能性**。编译器会强制你处理所有可能的情况。

### panic! 宏：不可恢复错误的最后手段

当程序遇到**不可恢复的严重错误**时，Rust 提供了 `panic!` 宏。调用 `panic!` 会导致程序立即终止，并打印错误信息和调用栈。

```rust
fn main() {
    panic!("程序遇到了无法恢复的错误！");
}
```

重要的是要理解：**panic 不是异常**！它不应该被用来处理正常的业务逻辑错误，而应该保留给那些真正表明程序处于无效状态的情况。

### ? 操作符：错误传播的语法糖

`?` 操作符是 Rust 中处理 `Result` 和 `Option` 的强大工具。它的作用是：

- 如果值是 `Ok(v)` 或 `Some(v)`，则解包并返回 `v`
- 如果值是 `Err(e)` 或 `None`，则提前返回函数，并将错误传播给调用者

```rust
fn read_username_from_file() -> Result<String, io::Error> {
    let mut file = File::open("username.txt")?; // 如果打开失败，立即返回 Err
    let mut username = String::new();
    file.read_to_string(&mut username)?; // 如果读取失败，立即返回 Err
    Ok(username)
}
```

这比传统的嵌套 match 表达式简洁得多！

## 代码演示：错误处理模式示例

让我们通过几个实际的例子来看看这些概念是如何工作的。

### 基本的 Result 处理

```rust
// example-01.rs 片段
fn divide(a: i32, b: i32) -> Result<i32, String> {
    if b == 0 {
        Err("除数不能为零".to_string())
    } else {
        Ok(a / b)
    }
}

fn main() {
    // 使用 match 处理 Result
    match divide(10, 2) {
        Ok(result) => println!("10 ÷ 2 = {}", result),
        Err(e) => println!("错误: {}", e),
    }
    
    match divide(10, 0) {
        Ok(result) => println!("10 ÷ 0 = {}", result),
        Err(e) => println!("错误: {}", e), // 这里会被执行
    }
}
```

### 使用 ? 操作符简化错误处理

```rust
use std::fs;
use std::io;

fn read_number_from_file(filename: &str) -> Result<i32, Box<dyn std::error::Error>> {
    let contents = fs::read_to_string(filename)?; // 自动传播 IO 错误
    let number: i32 = contents.trim().parse()?;   // 自动传播解析错误
    Ok(number)
}
```

注意这里我们使用了 `Box<dyn std::error::Error>` 作为错误类型，这是一种简单的错误类型擦除方式，适用于原型开发。

## 深入理解：错误类型设计与转换

### 自定义错误类型

在复杂的项目中，我们需要更精细的错误处理。这时就需要自定义错误类型：

```rust
// example-02.rs 片段
#[derive(Debug)]
enum ConfigError {
    IoError(io::Error),
    ParseError(ParseIntError),
    ValidationError(String),
}

impl std::error::Error for ConfigError {}
impl fmt::Display for ConfigError {
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        match self {
            ConfigError::IoError(e) => write!(f, "IO 错误: {}", e),
            ConfigError::ParseError(e) => write!(f, "解析错误: {}", e),
            ConfigError::ValidationError(msg) => write!(f, "验证错误: {}", msg),
        }
    }
}
```

### 错误转换：From trait

为了让 `?` 操作符能够自动转换错误类型，我们需要实现 `From` trait：

```rust
impl From<io::Error> for ConfigError {
    fn from(error: io::Error) -> Self {
        ConfigError::IoError(error)
    }
}

impl From<ParseIntError> for ConfigError {
    fn from(error: ParseIntError) -> Self {
        ConfigError::ParseError(error)
    }
}
```

现在，当我们使用 `?` 操作符时，标准库的错误会自动转换为我们的自定义错误类型！

### thiserror crate：简化错误实现

对于更复杂的项目，推荐使用 `thiserror` crate 来简化错误实现：

```rust
use thiserror::Error;

#[derive(Error, Debug)]
pub enum MyError {
    #[error("配置文件不存在: {0}")]
    NotFound(String),
    
    #[error("解析失败: {source}")]
    Parse {
        #[from]
        source: serde_json::Error,
    },
    
    #[error("网络错误: {0}")]
    Network(String),
}
```

## 实际应用：文件读取、网络请求错误处理

### 文件操作错误处理

```rust
use std::fs::File;
use std::io::{self, Read};

fn read_config_file(path: &str) -> Result<String, io::Error> {
    let mut file = File::open(path)?;
    let mut contents = String::new();
    file.read_to_string(&mut contents)?;
    Ok(contents)
}

// 使用示例
match read_config_file("config.json") {
    Ok(config) => println!("配置内容: {}", config),
    Err(e) => eprintln!("读取配置文件失败: {}", e),
}
```

### 网络请求错误处理

```rust
use reqwest;
use serde::{Deserialize, Serialize};

#[derive(Debug)]
enum ApiError {
    RequestError(reqwest::Error),
    JsonError(serde_json::Error),
    HttpError(u16),
}

async fn fetch_user_data(user_id: u32) -> Result<User, ApiError> {
    let response = reqwest::get(&format!("https://api.example.com/users/{}", user_id))
        .await
        .map_err(ApiError::RequestError)?;
    
    if !response.status().is_success() {
        return Err(ApiError::HttpError(response.status().as_u16()));
    }
    
    let user: User = response
        .json()
        .await
        .map_err(ApiError::JsonError)?;
    
    Ok(user)
}
```

## 常见误区：到处 unwrap、panic 不是异常

### 误区 1：滥用 unwrap()

新手常常这样写代码：

```rust
// ❌ 危险的做法！
let file = File::open("important.txt").unwrap();
let config = parse_config(&contents).unwrap();
let user_id = env::var("USER_ID").unwrap().parse::<u32>().unwrap();
```

这样做会在运行时导致程序崩溃，而且错误信息不够清晰。

### 正确做法：明确处理错误

```rust
// ✅ 正确的做法
match File::open("important.txt") {
    Ok(file) => {
        // 处理文件
    }
    Err(e) => {
        eprintln!("无法打开文件: {}", e);
        return Err(MyError::FileNotFound);
    }
}
```

或者使用 `?` 操作符：

```rust
// ✅ 更简洁的做法
let file = File::open("important.txt")?;
let config = parse_config(&contents)?;
let user_id: u32 = env::var("USER_ID")?.parse()?;
```

### 误区 2：认为 panic 是异常

在其他语言中，异常可以被捕获和处理。但在 Rust 中，**panic 不是用来处理正常错误的**。虽然你可以使用 `catch_unwind` 来捕获 panic，但这应该只用于非常特殊的情况（比如防止 FFI 边界崩溃）。

### 何时可以使用 unwrap/expect？

- **确定不会失败的操作**：比如 `env::args().next()` 总是返回 `Some`
- **测试代码**：在单元测试中，你可以安全地 unwrap 已知有效的值
- **原型开发**：在快速验证想法时，可以暂时使用 unwrap
- **使用 expect 提供更好的错误信息**：`map.get("key").expect("配置中必须包含 'key'")`

## 小结与思考

今天我们学习了 Rust 的错误处理机制，这是 Rust 安全性的重要组成部分。让我们回顾一下关键点：

### 核心原则
1. **明确性**：Rust 强制你在编译时考虑错误情况
2. **零成本抽象**：Result 和 Option 在运行时没有额外开销
3. **组合性**：? 操作符让错误处理代码既简洁又安全

### 最佳实践
- **优先使用 Result 而不是 panic**：除非是真正的不可恢复错误
- **设计良好的错误类型**：为不同的错误情况提供清晰的分类
- **合理使用 ? 操作符**：让它帮你传播错误，而不是隐藏错误
- **提供有用的错误信息**：使用 expect 而不是 unwrap，实现 Display trait

### 思考题
1. 为什么 Rust 选择不使用传统的异常处理机制？
2. 在什么情况下你应该创建自定义错误类型，而不是使用 `Box<dyn Error>`？
3. 如何设计一个既能提供详细错误信息，又便于调用者处理的错误类型？

记住，好的错误处理不仅仅是让程序不崩溃，更是提供清晰的反馈，帮助用户和开发者理解发生了什么问题，以及如何解决它。

就像一个好的天气预报不仅能告诉你是否会下雨，还能告诉你雨量大小、持续时间，甚至建议你带伞还是改期出行。Rust 的错误处理机制给了我们同样的能力——让我们能够构建既安全又用户友好的软件系统。

继续练习，你会发现错误处理不再是编程中的负担，而是构建可靠软件的有力工具！
