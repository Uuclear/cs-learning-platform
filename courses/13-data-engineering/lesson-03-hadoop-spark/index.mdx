## 开场白

想象一下，你是一家大型电商平台的数据工程师。每天，你的系统需要处理超过 **10亿条** 用户行为日志，包括页面浏览、商品点击、购物车添加和订单完成等操作。这些日志数据总量达到 **TB级别**，传统的单机数据库和处理工具根本无法在合理时间内完成分析任务。

就在几年前，这样的数据处理需求几乎是不可能完成的任务。但随着 **Hadoop** 和 **Spark** 等大数据框架的出现，分布式计算变得触手可及。今天，我们将一起探索这些革命性技术如何改变了数据处理的格局，以及它们背后的原理和实践方法。

通过本课程，你将理解为什么大数据不仅仅是"更多的数据"，而是一种全新的计算范式，能够处理传统方法无法应对的 **Volume（体量）**、**Velocity（速度）** 和 **Variety（多样性）** 挑战。

## 核心概念

### MapReduce 编程模型

MapReduce 是 Google 提出的一种编程模型，也是 Hadoop 的核心计算引擎。它将复杂的数据处理任务分解为两个简单的阶段：

- **Map 阶段**：将输入数据分割成键值对 `(key, value)`，并对每个键值对执行用户定义的 map 函数
- **Reduce 阶段**：将具有相同 key 的所有 value 聚合在一起，并执行用户定义的 reduce 函数

MapReduce 的优势在于其 **自动并行化** 和 **容错性**。框架会自动处理数据分区、任务调度、节点故障恢复等复杂问题。

### HDFS（Hadoop 分布式文件系统）

HDFS 是 Hadoop 的存储层，专为大规模数据存储而设计：

- **高容错性**：数据自动复制到多个节点（默认3份）
- **高吞吐量**：优化了批量数据访问，而非低延迟访问
- **大文件友好**：适合存储 GB 到 TB 级别的大文件
- **一次写入多次读取**：文件一旦创建就不支持修改

### Spark 架构

Apache Spark 是一个更快、更通用的大数据处理引擎，其核心组件包括：

- **RDD（弹性分布式数据集）**：不可变的、分区的元素集合，支持并行操作
- **DAG（有向无环图）执行引擎**：将多个操作优化为单个执行计划，减少磁盘 I/O
- **内存计算**：尽可能将数据保存在内存中，避免频繁的磁盘读写
- **统一 API**：支持批处理、流处理、机器学习和图计算

Spark 的关键创新在于 **惰性求值** 和 **血统（Lineage）** 机制，使得错误恢复更加高效。

## 代码演示

让我们通过三个实际的代码示例来理解这些概念：

### 示例 1：MapReduce 词频统计模拟

```python
# example-01-mapreduce-simulation.py
def map_function(document_id, text):
    """Map 函数：将文档分解为 (单词, 1) 键值对"""
    words = re.findall(r'\b\w+\b', text.lower())
    return [(word, 1) for word in words]

def reduce_function(word, counts):
    """Reduce 函数：聚合相同单词的计数"""
    return (word, sum(counts))
```

这个示例展示了 MapReduce 的基本工作流程：Map 阶段生成单词计数，Shuffle 阶段按键分组，Reduce 阶段计算总和。

### 示例 2：Spark RDD 操作模拟

```python
# example-02-spark-rdd.py
class RDD:
    def map(self, func):
        """转换操作：对每个元素应用函数"""
        mapped_data = [func(item) for item in self.data]
        return RDD(mapped_data)
    
    def reduce(self, func):
        """行动操作：聚合所有元素"""
        result = self.data[0]
        for item in self.data[1:]:
            result = func(result, item)
        return result
```

这个 RDD 模拟展示了 Spark 的核心抽象：**转换操作**（返回新的 RDD）和 **行动操作**（触发实际计算并返回结果）。

## 深入理解

### Spark 执行模型

Spark 的执行模型基于 **DAG（有向无环图）**，这与 MapReduce 的固定两阶段模型形成鲜明对比：

1. **逻辑执行计划**：用户代码被转换为逻辑 DAG
2. **物理执行计划**：逻辑 DAG 被优化并转换为物理执行计划
3. **任务调度**：物理计划被分解为多个 Stage，每个 Stage 包含多个 Task
4. **执行**：Task 在集群中的 Executor 上并行执行

这种模型允许 Spark 进行 **流水线优化**，将多个转换操作合并到同一个 Stage 中，减少数据落盘。

### 分区和洗牌（Shuffling）

**分区（Partitioning）** 是分布式计算的核心概念：
- 数据被分割成多个分区，每个分区可以在不同节点上独立处理
- 合理的分区策略可以最大化并行度并最小化网络传输

**洗牌（Shuffling）** 是代价最高的操作之一：
- 当操作需要重新组织数据（如 groupByKey、join）时发生
- 涉及大量的网络 I/O 和磁盘 I/O
- 应该尽量避免或优化洗牌操作

### 宽依赖 vs 窄依赖

- **窄依赖**：父 RDD 的每个分区最多被子 RDD 的一个分区使用（如 map、filter）
  - 可以在单个 Stage 中流水线执行
  - 故障恢复只需要重新计算丢失的分区

- **宽依赖**：父 RDD 的每个分区可能被子 RDD 的多个分区使用（如 groupByKey、reduceByKey）
  - 需要洗牌操作，形成新的 Stage 边界
  - 故障恢复需要重新计算整个父 RDD

## 实际应用

### 日志分析管道

在实际生产环境中，日志分析是大数据处理的典型应用场景：

1. **数据收集**：从 Web 服务器、应用程序收集日志
2. **数据清洗**：过滤无效记录、标准化格式
3. **用户行为分析**：统计页面浏览量、用户路径、转化率
4. **异常检测**：识别错误模式、安全威胁
5. **实时监控**：构建仪表板，实时显示关键指标

使用 Spark，我们可以轻松构建这样的 pipeline：

```python
logs_df = spark.read.json("hdfs://logs/")
cleaned_logs = logs_df.filter(logs_df.status_code == 200)
page_views = cleaned_logs.groupBy("page").count()
page_views.write.parquet("hdfs://analytics/page_views/")
```

### Word Count 应用

虽然 Word Count 看起来简单，但它代表了许多实际问题：

- **搜索引擎**：统计网页关键词频率
- **推荐系统**：分析用户兴趣标签
- **情感分析**：统计正面/负面词汇出现频率
- **数据质量检查**：识别异常模式或重复内容

### 数据转换和 ETL

大数据处理的另一个重要应用是 ETL（Extract, Transform, Load）：

- **数据提取**：从各种源系统（数据库、API、文件）提取数据
- **数据转换**：清洗、聚合、丰富数据
- **数据加载**：将处理后的数据加载到目标系统（数据仓库、BI 工具）

Spark 的 DataFrame API 特别适合这类任务，提供了类似 SQL 的简洁接口。

## 常见误区

### 误区 1：Spark 总是比 Hadoop 快

虽然 Spark 在内存计算方面确实更快，但这并不意味着它在所有场景下都优于 Hadoop：
- **内存限制**：如果数据量超过可用内存，Spark 的性能会急剧下降
- **成本考虑**：内存比磁盘昂贵，大规模部署的成本更高
- **特定工作负载**：对于简单的批处理任务，Hadoop MapReduce 可能更稳定和经济

### 误区 2：RDD 比 DataFrame 更灵活

实际上，DataFrame/Dataset API 通常比 RDD 更优：
- **优化器**：Catalyst 优化器可以自动优化查询计划
- **序列化**：Tungsten 引擎提供高效的内存管理和序列化
- **类型安全**：Dataset 提供编译时类型检查
- **性能**：通常比同等的 RDD 操作快 2-10 倍

### 误区 3：更多分区总是更好

分区数量需要根据实际情况权衡：
- **太少分区**：无法充分利用集群资源，并行度不足
- **太多分区**：任务调度开销过大，小文件问题严重
- **最佳实践**：每个分区大小在 100MB-1GB 之间，分区数约为集群 CPU 核心数的 2-3 倍

### 误区 4：大数据技术可以解决所有数据问题

大数据框架并非万能药：
- **小数据场景**：传统数据库和单机工具可能更简单高效
- **实时性要求**：某些场景需要专门的流处理框架（如 Flink、Kafka Streams）
- **复杂事务**：大数据系统通常不支持 ACID 事务
- **学习成本**：维护和调优大数据集群需要专业知识

## 小结与思考

### 关键要点回顾

1. **MapReduce** 提供了简单而强大的分布式计算模型，但存在磁盘 I/O 瓶颈
2. **Spark** 通过内存计算和 DAG 执行引擎显著提升了性能
3. **RDD** 是 Spark 的核心抽象，区分转换操作和行动操作至关重要
4. **分区和洗牌** 是影响性能的关键因素，需要仔细设计
5. **实际应用** 包括日志分析、ETL、机器学习等多个领域
