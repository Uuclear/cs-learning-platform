# 挑战 2：数据湖模式演进实战

## 背景
你负责维护一个用户数据湖，当前存储了数百万用户的个人信息。初始模式如下：

```json
{
  "user_id": "integer",
  "name": "string", 
  "email": "string",
  "age": "integer",
  "country": "string",
  "registration_date": "date"
}
```

业务需求不断变化，需要进行以下模式变更：

1. **添加新字段**：`phone_number` (string, 可选), `is_premium` (boolean, 默认false)
2. **修改字段类型**：将 `age` 从 integer 改为 date_of_birth (date)
3. **移除字段**：移除 `country` 字段（迁移到单独的用户偏好表）
4. **添加复杂字段**：`preferences` (object, 包含主题、语言等嵌套字段)

## 任务要求

### 1. 兼容性分析
对每个变更进行兼容性分析：

- 哪些变更是向后兼容的？
- 哪些变更是前向兼容的？
- 哪些变更会导致不兼容？
- 对现有数据消费者的影响是什么？

### 2. 实施策略
为每个变更设计实施策略：

- **向后兼容变更**：如何安全地添加新字段？
- **不兼容变更**：如何处理 `age` 到 `date_of_birth` 的迁移？
- **字段移除**：如何确保依赖 `country` 字段的应用程序不会失败？
- **数据迁移**：是否需要批量数据迁移？如果需要，如何执行？

### 3. 验证方案
设计验证方案确保模式变更成功：

- 如何验证新旧数据都能被正确读取？
- 如何测试现有应用程序的兼容性？
- 如何监控模式变更后的数据质量问题？

### 4. 回滚计划
为每个高风险变更设计回滚计划：

- 如果变更导致问题，如何快速回滚？
- 需要保留哪些历史数据以支持回滚？
- 回滚的时间窗口是多长？

## 提交要求
提交一份详细的模式演进实施计划，包含兼容性矩阵、实施步骤、验证方案和回滚策略。同时提供一个Python脚本模拟关键的模式变更操作。
