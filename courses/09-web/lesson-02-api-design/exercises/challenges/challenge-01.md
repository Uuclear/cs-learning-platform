# 图书管理API设计

**难度**: ⭐⭐⭐

## 描述
设计并实现一个完整的图书管理API系统，用于图书馆或书店的后端服务。这个API需要支持图书的基本管理功能，同时考虑实际业务需求和最佳实践。

## 需求

### 核心功能
- **图书管理**: 支持图书的增删改查操作
  - 图书信息包括: 书名、作者、ISBN、出版社、出版日期、分类、价格、库存数量
  - ISBN必须唯一且符合标准格式验证
  - 支持按多种条件搜索和过滤图书

- **借阅/购买管理**: 
  - 用户可以借阅图书（图书馆场景）或购买图书（书店场景）
  - 记录交易历史，包括借阅/购买时间、归还/退货时间（如果适用）
  - 库存管理：借阅/购买时减少库存，归还/退货时增加库存

- **用户管理**:
  - 用户注册和认证（JWT）
  - 用户角色：普通用户、管理员
  - 管理员可以管理图书和查看所有用户的借阅/购买记录

### API设计要求
- **RESTful设计**: 遵循REST原则，使用合适的HTTP方法和状态码
- **分页支持**: 图书列表和交易记录需要支持分页
- **错误处理**: 统一的错误响应格式，包含错误代码和描述
- **输入验证**: 所有输入都需要进行验证，包括数据类型、格式、范围等
- **安全性**: 
  - 敏感操作需要认证和授权
  - 密码需要安全存储（哈希）
  - 防止常见安全漏洞（如SQL注入、XSS等）

### 高级功能（可选）
- **图书推荐**: 基于用户历史行为的简单推荐
- **库存预警**: 当库存低于阈值时自动提醒
- **数据导出**: 支持将图书或交易数据导出为CSV格式
- **速率限制**: 防止API滥用

## API端点设计示例

```
# 认证
POST   /api/v1/auth/register
POST   /api/v1/auth/login

# 图书管理
GET    /api/v1/books                    # 获取图书列表（支持分页/过滤）
POST   /api/v1/books                    # 创建新图书（管理员）
GET    /api/v1/books/{id}               # 获取单本图书详情
PUT    /api/v1/books/{id}               # 更新图书信息（管理员）
DELETE /api/v1/books/{id}               # 删除图书（管理员）

# 借阅/购买
POST   /api/v1/transactions/borrow       # 借阅图书
POST   /api/v1/transactions/return       # 归还图书
POST   /api/v1/transactions/purchase     # 购买图书（如果是书店场景）

# 用户相关
GET    /api/v1/users/me                 # 获取当前用户信息
GET    /api/v1/users/{id}/transactions  # 获取用户的交易记录（自己的或管理员查看他人）
```

## 提示

1. **数据库设计**: 先设计好数据模型，考虑表之间的关系（一对多、多对多等）

2. **验证逻辑**: 
   - ISBN验证可以使用正则表达式
   - 日期格式验证
   - 数值范围验证（价格>0，库存>=0等）

3. **事务处理**: 借阅/购买操作涉及多个步骤（减少库存、创建交易记录），需要考虑原子性

4. **测试用例**: 编写测试用例覆盖正常情况和各种错误情况

5. **文档**: 使用OpenAPI/Swagger规范编写API文档

6. **技术栈建议**: 
   - 后端: Python (Flask/FastAPI) 或 Node.js (Express)
   - 数据库: SQLite（开发）或 PostgreSQL/MySQL（生产）
   - 认证: JWT
   - 验证: 使用专门的验证库

## 评估标准

- ✅ 错误处理是否完善
- ✅ 安全性考虑是否充分
- ✅ 代码结构是否清晰可维护
- ✅ 功能是否完整实现需求
- ✅ 性能和扩展性考虑
