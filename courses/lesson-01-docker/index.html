11-01 Docker容器化：从开发到部署

开场白 - 从"在我机器上能跑"到容器化的诞生

你是否曾经遇到过这样的场景？开发同事兴奋地说："我的代码完美运行！"，但当你尝试在自己的机器上运行时，却遇到了各种依赖冲突、版本不兼容的问题。这就是经典的"在我机器上能跑"（It works on my machine）问题。

容器化技术正是为了解决这类环境一致性问题而诞生的。就像现实世界中的集装箱一样，Docker 容器将应用程序及其所有依赖打包成一个标准化的单元，确保应用在任何环境中都能一致地运行。🚀

核心概念 - 什么是容器、容器vs虚拟机、Docker架构

什么是容器？

容器是一个轻量级、可移植的软件单元，它包含了运行应用程序所需的所有内容：代码、运行时、系统工具、系统库和配置文件。

容器 vs 虚拟机

特性 虚拟机 (VM) 容器
隔离级别 硬件级隔离 操作系统级隔离
启动时间 分钟级 秒级
资源占用 高（需要完整OS） 低（共享宿主机OS内核）
密度 低 高

Docker 架构三大核心组件

  1. 镜像 (Image) - 只读模板，包含创建容器所需的指令
  2. 容器 (Container) - 镜像的运行实例
  3. 仓库 (Registry) - 存储和分发镜像的地方（如 Docker Hub）

代码演示

让我们通过三个实际例子来理解 Docker 的核心概念：

示例 1: 基础 Dockerfile

# 使用官方 Python 运行时作为基础镜像
FROM python:3.9-slim

# 设置工作目录
WORKDIR /app

# 复制依赖文件
COPY requirements.txt .

# 安装依赖
RUN pip install --no-cache-dir -r requirements.txt

# 复制应用代码
COPY . .

# 暴露端口
EXPOSE 5000

# 运行应用
CMD ["python", "app.py"]

示例 2: 多阶段构建

# 构建阶段
FROM node:16 as builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production

# 运行阶段
FROM node:16-alpine
WORKDIR /app
COPY --from=builder /app/node_modules ./node_modules
COPY . .
CMD ["node", "server.js"]

示例 3: Docker Compose

version: '3.8'
services:
  web:
    build: .
    ports:
      - "5000:5000"
    depends_on:
      - redis
  redis:
    image: "redis:alpine"

深入理解 - Dockerfile最佳实践、多阶段构建、数据卷与网络

Dockerfile 最佳实践

  • 使用特定标签：避免使用 latest，使用具体版本如 python:3.9-slim
  • 最小化层数：合并相关的 RUN 指令
  • .dockerignore：排除不必要的文件
  • 安全考虑：使用非 root 用户运行应用

多阶段构建 (Multi-stage Builds)

多阶段构建允许我们在一个 Dockerfile 中使用多个 FROM 指令，每个 FROM 指令可以使用不同的基础镜像，并且可以选择性地将文件从一个阶段复制到另一个阶段。这有助于创建更小、更安全的最终镜像。

数据卷与网络

  • 数据卷 (Volumes)：用于持久化数据，独立于容器生命周期
  • 绑定挂载 (Bind Mounts)：将主机目录直接挂载到容器
  • 网络 (Networks)：容器间通信的基础，支持自定义网络隔离

实际应用 - 用Docker部署一个完整的Web应用（Python+Redis）

让我们部署一个简单的投票应用，包含 Flask Web 应用和 Redis 数据库：

  1. 创建应用代码 (app.py)
from flask import Flask, render_template, request, redirect
import redis

app = Flask(__name__)
r = redis.Redis(host='redis', port=6379, decode_responses=True)

@app.route('/')
def index():
    return render_template('index.html', votes=r.get('votes') or 0)

@app.route('/vote', methods=['POST'])
def vote():
    r.incr('votes')
    return redirect('/')
  1. 创建 Docker Compose 文件
version: '3'
services:
  web:
    build: .
    ports:
      - "5000:5000"
    depends_on:
      - redis
  redis:
    image: "redis:alpine"
  1. 构建并运行
docker-compose up --build

常见误区

❌ 容器不是虚拟机

容器共享宿主机的操作系统内核，而不是运行完整的操作系统。它们更轻量、启动更快。

❌ 每个容器一个进程

遵循单一职责原则，每个容器应该只运行一个主进程。这使得容器更容易管理、扩展和调试。

❌ 镜像不是越大越好

较小的镜像意味着更快的传输速度、更少的安全漏洞和更低的资源消耗。使用 Alpine Linux 或其他精简基础镜像。

小结与思考

Docker 容器化技术彻底改变了我们开发、测试和部署应用程序的方式。通过将应用及其依赖打包成标准化的容器，我们实现了真正的"一次构建，到处运行"。

思考题：

  1. 在什么场景下，容器化可能不是最佳选择？
  2. 如何处理容器中的敏感信息（如 API 密钥）？
  3. 容器编排工具（如 Kubernetes）解决了哪些 Docker Compose 无法解决的问题？

记住，容器化不仅仅是技术，更是一种思维方式——关注可移植性、一致性和效率。🎯
