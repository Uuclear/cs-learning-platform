## 开场白

在现代云原生应用开发中，我们经常面临一个挑战：如何从管理几个容器扩展到管理成百上千个容器？当应用规模增长时，手动管理容器变得几乎不可能。Kubernetes（简称 K8s）正是为了解决这个规模化问题而诞生的开源容器编排平台。

想象一下，你的应用需要处理突发流量，可能需要动态扩展从 2 个实例到 50 个实例；或者当某个容器崩溃时，系统需要自动重启它；又或者你需要无缝地进行应用更新而不影响用户。这些复杂的操作在 Kubernetes 中都变得简单和自动化。

Kubernetes 最初由 Google 开发，现在由 Cloud Native Computing Foundation (CNCF) 维护，已经成为容器编排的事实标准。它提供了一个声明式的 API，让你可以描述你想要的状态，而 Kubernetes 会确保实际状态与期望状态保持一致。

## 核心概念

Kubernetes 的架构基于几个核心概念，理解这些概念是掌握 Kubernetes 的关键：

### Pod
Pod 是 Kubernetes 中最小的部署单元，它可以包含一个或多个紧密相关的容器。同一个 Pod 中的容器共享网络命名空间、存储卷和生命周期。通常情况下，一个 Pod 只运行一个主容器，但在某些场景下（如 sidecar 模式），可能会有多个容器。

### Service
Service 为一组 Pod 提供稳定的网络端点和负载均衡。即使后端的 Pod 被重新调度或替换，Service 的 IP 和 DNS 名称保持不变。Service 支持多种类型：ClusterIP（集群内部访问）、NodePort（通过节点端口暴露）、LoadBalancer（云提供商负载均衡器）和 ExternalName（外部服务别名）。

### Deployment
Deployment 是管理 Pod 副本的高级抽象，它确保指定数量的 Pod 副本始终运行，并支持滚动更新、回滚等操作。通过 Deployment，你可以声明应用的期望状态，Kubernetes 会自动处理扩缩容和更新过程。

### ConfigMap 和 Secret
ConfigMap 用于存储非敏感的配置数据，Secret 用于存储敏感信息（如密码、token）。它们都可以通过环境变量或挂载卷的方式注入到容器中，实现配置与代码的分离。

### Namespace
Namespace 提供了虚拟集群的隔离机制，允许在同一物理集群中创建多个逻辑上隔离的环境（如 dev、staging、prod）。

### Ingress
Ingress 是管理外部 HTTP/HTTPS 访问集群内服务的规则集合。它通常与 Ingress Controller 配合使用，提供基于路径或主机名的路由、SSL 终止等功能。

### 示例 1：用 Python 生成 Deployment / Service / Ingress

手工写 YAML 容易 copy-paste 出错。下面用字典描述期望状态，再 `yaml.dump` 输出——和 Helm/Kustomize 的思路一致：

```python
def generate_deployment_manifest(app_name: str, image: str, replicas: int = 2):
    return {
        "apiVersion": "apps/v1",
        "kind": "Deployment",
        "metadata": {"name": f"{app_name}-deployment", "labels": {"app": app_name}},
        "spec": {
            "replicas": replicas,
            "selector": {"matchLabels": {"app": app_name}},
            "template": {
                "metadata": {"labels": {"app": app_name}},
                "spec": {
                    "containers": [{
                        "name": app_name,
                        "image": image,
                        "resources": {
                            "requests": {"cpu": "100m", "memory": "128Mi"},
                            "limits": {"cpu": "200m", "memory": "256Mi"},
                        },
                    }]
                },
            },
        },
    }
```

CI 流水线里可根据环境变量生成不同副本数、镜像 tag，避免「生产误用 latest」。

### 示例 2：滚动更新 vs 重新创建

`example-02-deployment-strategies.py` 模拟两种策略下**可用 Pod 数量**随时间变化：滚动更新保持 `maxUnavailable` 以内的实例在线；Recreate 会先杀光旧 Pod 再拉起新版——短暂停机，但资源占用更简单。

### 示例 3：HPA 如何根据 CPU 扩缩容

`example-03-hpa-simulation.py` 用循环读取「模拟 CPU 指标」，按目标利用率计算 `desired_replicas = ceil(current * (current_cpu / target_cpu))`，展示 HPA 控制器的大致决策逻辑（真实集群由 Metrics Server + HPA 对象完成）。

## 深入理解

### 调度器工作原理
Kubernetes 调度器负责将 Pod 分配到合适的节点上。它通过两个阶段工作：过滤（Filtering）和评分（Scoring）。过滤阶段排除不符合 Pod 要求的节点（如资源不足、节点选择器不匹配），评分阶段为剩余节点打分并选择最优节点。

### HPA 自动伸缩
HPA 基于 CPU 使用率、内存使用率或其他自定义指标自动调整 Deployment 或 ReplicaSet 的副本数。它通过定期查询 Metrics Server 获取指标数据，并根据预设的目标值计算新的副本数。

### 滚动更新
滚动更新策略允许在不中断服务的情况下更新应用。Kubernetes 会逐步创建新版本的 Pod，同时逐步删除旧版本的 Pod。通过设置 `maxUnavailable` 和 `maxSurge` 参数，可以控制更新过程中的可用性和资源使用。

### 健康检查
Kubernetes 提供两种健康检查探针：
- **Liveness Probe**：检测容器是否仍在运行，失败时会重启容器
- **Readiness Probe**：检测容器是否准备好接收流量，失败时会从 Service 的负载均衡池中移除

## 常见误区

### K8s 不是万能的
Kubernetes 是一个强大的工具，但它并不适合所有场景。对于简单的单体应用或小型项目，使用 Kubernetes 可能会增加不必要的复杂性。

### 资源限制很重要
如果不为容器设置合理的资源请求（requests）和限制（limits），可能会导致资源争用、节点不稳定或 Pod 被意外终止。

### 不要在生产环境使用 latest 标签
使用 `latest` 标签会使部署变得不可预测和不可重现。始终使用明确的版本标签，如 `v1.2.3`。

### 忽视网络策略
默认情况下，Kubernetes 中的所有 Pod 都可以相互通信。在生产环境中，应该使用 NetworkPolicy 来限制不必要的网络连接，提高安全性。
