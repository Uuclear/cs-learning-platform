# 实现简单查询优化器

⭐⭐⭐⭐⭐

## 描述

现代数据库的查询优化器是极其复杂的组件，但其核心思想可以简化理解。你的任务是实现一个简化的查询优化器，能够为给定的 SQL 查询生成多个可能的执行计划，并选择成本最低的一个。

这个挑战将帮助你深入理解查询优化器的工作原理，包括成本估算、统计信息收集和计划选择。

## 要求

1. **统计信息收集**：
   - 实现表行数统计
   - 实现列的唯一值数量统计
   - 实现简单的直方图（可选）

2. **成本模型实现**：
   - 全表扫描成本 = 表行数 × 常数
   - 索引扫描成本 = log(表行数) + 返回行数 × 常数
   - 嵌套循环连接成本 = 外表行数 × 内表扫描成本
   - 索引连接成本 = 外表行数 × 索引查找成本

3. **执行计划生成**：
   - 为单表查询生成全表扫描和索引扫描计划
   - 为两表连接生成嵌套循环和索引连接计划
   - 支持 WHERE 条件的选择性估算

4. **查询解析**：
   - 解析简单的 SELECT 查询（支持 WHERE、JOIN、GROUP BY）
   - 提取表名、列名和条件表达式
   - 识别 JOIN 条件和过滤条件

5. **优化器核心**：
   - 基于统计信息估算每个计划的成本
   - 选择成本最低的执行计划
   - 输出详细的执行计划说明

## 测试用例

你的优化器应该能够正确处理以下查询：

```sql
-- 单表查询
SELECT * FROM users WHERE email = 'user@example.com';
SELECT * FROM products WHERE price BETWEEN 100 AND 500;

-- 连接查询  
SELECT u.name, o.amount FROM users u JOIN orders o ON u.id = o.user_id WHERE o.status = 'delivered';
SELECT p.name, COUNT(*) FROM products p JOIN orders o ON p.id = o.product_id GROUP BY p.name;
```

## 提示

- **简化假设**：假设数据均匀分布，选择性 = 1/唯一值数量
- **索引检测**：通过查询 SQLite 的 `sqlite_master` 表获取索引信息
- **成本常数**：I/O 成本设为 1.0，CPU 成本设为 0.1
- **递归处理**：对于多表连接，使用动态规划或贪心算法选择连接顺序
- **缓存统计信息**：避免重复计算统计信息

## 额外挑战

- 支持三表或更多表的连接优化
- 实现简单的谓词下推优化
- 添加对 LIMIT 子句的优化支持
- 与实际的 SQLite 执行时间进行对比验证
