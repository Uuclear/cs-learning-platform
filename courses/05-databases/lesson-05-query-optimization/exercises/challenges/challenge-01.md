# 诊断和优化慢查询

⭐⭐⭐⭐

## 描述

你是一家电商公司的数据库工程师，最近收到用户投诉说商品搜索和订单查询功能变得非常慢。通过监控系统发现，有几个关键查询的执行时间从原来的几十毫秒增加到了几秒钟。

你的任务是分析这些慢查询，找出性能瓶颈，并提供优化方案。

## 要求

1. **数据准备**：使用提供的脚本创建一个包含 50,000 个产品和 200,000 个订单的测试数据库
2. **问题识别**：使用 `EXPLAIN QUERY PLAN` 分析以下三个慢查询：
   - 商品搜索查询：`SELECT * FROM products WHERE name LIKE '%手机%' OR description LIKE '%智能手机%'`
   - 订单统计查询：`SELECT p.category, COUNT(*), SUM(o.amount) FROM products p, orders o WHERE p.id = o.product_id GROUP BY p.category`
   - 用户订单查询：`SELECT * FROM orders WHERE strftime('%Y', order_date) = '2023' AND customer_id = 12345`
3. **优化实施**：
   - 为每个查询创建合适的索引
   - 重写查询语句以避免索引失效
   - 对于全文搜索需求，考虑使用 SQLite 的 FTS（全文搜索）功能
4. **性能验证**：
   - 测量优化前后的执行时间
   - 确保优化后的查询返回相同的结果
   - 性能提升目标：至少 90% 的时间减少

## 提示

- **LIKE 查询优化**：通配符开头的 LIKE 无法使用普通索引，考虑使用 FTS 或反向索引
- **日期函数优化**：避免在列上使用函数，改为使用范围查询
- **隐式 JOIN 优化**：显式使用 JOIN 语法，并确保连接条件有索引
- **复合索引设计**：根据查询模式设计合适的复合索引顺序
- **统计信息更新**：在创建新索引后，SQLite 会自动更新统计信息

## 额外挑战

- 实现一个自动化的慢查询检测工具，能够扫描 SQL 日志并标记潜在的性能问题
- 创建一个性能基准测试，确保未来的代码变更不会引入回归

记住：最好的优化往往是避免执行不必要的查询。考虑是否可以在应用层面添加缓存或预计算。
