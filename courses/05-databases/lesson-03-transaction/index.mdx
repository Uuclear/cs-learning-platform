# 05-03 事务：要么全做，要么全不做

想象一下这样的场景：你正在银行进行一笔转账，把1000元从账户A转到账户B。系统开始执行转账操作——首先从账户A扣除了1000元，但就在准备给账户B增加1000元的时候，突然停电了！现在钱从A账户消失了，但B账户却没收到，这笔钱去了哪里？

这就是为什么我们需要**数据库事务**——它确保一系列操作要么全部成功，要么全部失败回滚，绝不会出现"半成品"状态。

## 1. 开场白：为什么需要事务？

在现实世界中，很多操作都是不可分割的整体：
- 银行转账：扣款和存款必须同时成功或同时失败
- 订单处理：扣库存、创建订单、扣用户余额必须原子性完成  
- 航班预订：锁定座位、扣款、生成票务信息必须一致

如果没有事务保护，系统故障、网络中断或并发访问都可能导致数据不一致，造成严重的业务问题。

## 2. 核心概念：ACID 特性

数据库事务遵循 **ACID** 四大特性：

### Atomicity（原子性）
事务中的所有操作被视为一个不可分割的单元。要么全部执行成功，要么全部回滚，不存在中间状态。

```sql
BEGIN;           -- 开始事务
UPDATE accounts SET balance = balance - 1000 WHERE id = 'A';
UPDATE accounts SET balance = balance + 1000 WHERE id = 'B';
COMMIT;          -- 提交事务（全部成功）
-- 或者 ROLLBACK; -- 回滚事务（全部失败）
```

### Consistency（一致性）
事务必须保证数据库从一个一致性状态转换到另一个一致性状态。比如账户余额不能为负数，外键约束必须满足等。

### Isolation（隔离性）
多个并发事务之间相互隔离，一个事务的执行不应影响其他事务的执行结果。

### Durability（持久性）
一旦事务提交，其对数据库的修改就是永久性的，即使系统崩溃也不会丢失。

## 4. 深入理解：隔离级别

不同的隔离级别提供了不同程度的并发控制：

### Read Uncommitted（读未提交）
最低隔离级别，允许读取其他事务未提交的数据，可能导致**脏读**。

### Read Committed（读已提交）
只允许读取已提交的数据，避免脏读，但可能出现**不可重复读**。

### Repeatable Read（可重复读）
保证在同一事务中多次读取同一数据结果一致，避免不可重复读，但可能出现**幻读**。

### Serializable（串行化）
最高隔离级别，完全隔离并发事务，性能最差但最安全。

#### 并发异常类型：
- **脏读（Dirty Read）**：读取了其他事务未提交的数据
- **不可重复读（Non-repeatable Read）**：同一事务中多次读取同一数据得到不同结果
- **幻读（Phantom Read）**：同一查询条件在不同时间返回不同行数

## 5. 实际应用

在真实系统中，事务处理需要考虑：

### 并发控制策略
- **悲观锁**：假设冲突会发生，提前加锁
- **乐观锁**：假设冲突很少发生，只在提交时检查冲突

### 事务边界设计
- 尽量缩短事务持续时间
- 避免在事务中进行用户交互
- 合理选择隔离级别（通常Read Committed足够）

### 分布式事务
跨多个数据库或服务的事务需要更复杂的协调机制，如两阶段提交（2PC）。

## 6. 常见误区

### ❌ 误区1：事务能解决所有并发问题
事务主要解决数据一致性问题，但不能自动解决所有并发场景。需要结合适当的隔离级别和锁策略。

### ❌ 误区2：越高隔离级别越好
更高的隔离级别意味着更差的并发性能。应该根据业务需求选择合适的隔离级别。

### ❌ 误区3：长事务没问题
长时间运行的事务会持有锁，阻塞其他事务，导致性能问题甚至死锁。

### ❌ 误区4：自动提交模式下不需要显式事务
自动提交模式下每个语句都是独立事务，无法保证多个操作的原子性。

## 8. 小结与思考

事务是数据库系统的基石，它通过ACID特性保证了数据的一致性和可靠性。关键要点：

- **原子性**确保操作完整性
- **隔离级别**需要在一致性和性能间权衡
- **合理设计**事务边界和持续时间
- **理解并发异常**并选择合适的解决方案
